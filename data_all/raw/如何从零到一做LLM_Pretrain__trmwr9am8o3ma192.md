**<font style="color:#601bde;">1. 行业现状与自研的战略意义</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

当前，[dense 模型](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=dense+%E6%A8%A1%E5%9E%8B&zhida_source=entity)领域有 [qwen](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=qwen&zhida_source=entity) 表现优异，[MOE 模型](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=MOE+%E6%A8%A1%E5%9E%8B&zhida_source=entity)方向有 [deepseek](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=deepseek&zhida_source=entity) 领跑，而小尺寸模型中 [minicpm](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=minicpm&zhida_source=entity) 亦备受瞩目。对于个人开发者或大型企业而言，在同等参数规模下训练出超越这些头部开源模型的难度极高。尽管大模型预训练阶段全面拥抱开源的趋势日益明显，但自研预训练模型依然具备不可替代的价值。

从专业技术与商业逻辑角度分析，自研的意义主要体现在以下四个方面：

+ **突破“伪开源”限制，掌握核心迭代能力：** 目前多数企业仅开源了模型权重参数，而训练框架、数据配比等核心资产依然闭源。这导致使用者难以参与到模型底层的迭代优化中，开源方也面临商业转化率受损的风险。因此，在 [llm](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=llm&zhida_source=entity) 实现真正意义上的全面开源（即公开训练代码与数据配比，支持社区通过提交 CR 优化训练效率）之前，掌握完整的预训练技术栈仍是构建核心竞争力的关键。
+ **满足垂直领域持续预训练（Continue-Pretrain）需求：** 通用大模型的商业变现能力通常不及垂直领域（Domain）模型。随着 [continue-pretrain](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=continue-pretrain&zhida_source=entity) 需求的激增，掌握预训练技术显得尤为重要，因为两者的底层技术栈具有高度一致性。
+ **实现数据透明化，赋能对齐（Alignment）阶段：** 若非自主完成预训练，开发者将无法确切知晓模型在预训练阶段所吸收的数据分布与知识结构。仅凭常规评估难以精确衡量模型对各类数据的掌握程度。缺乏这些底层数据细节，在对齐阶段便无法进行针对性优化，从而难以最大限度地激发模型潜力。例如，在 SFT（监督微调）阶段要求一个未曾学习过古典诗词的通用模型创作唐诗，必然会导致严重的幻觉问题。
+ **实现 Tokenizer 与解码效率的自主可控：** 采用开源模型意味着必须接受其内置的 [tokenizer](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=tokenizer&zhida_source=entity)，这直接影响解码速度及业务适配性。例如，在意图识别任务中，若使用 LLaMA 模型，特定意图（如 `ai.listen.music`）可能被切分为 5 个 token；而自研模型则可将其定制为 1 个 token，从而显著提升生成效率。尽管扩充词表技术已相对成熟，但其不仅需要消耗额外算力以恢复模型性能，且难以保证在引入新语料后模型整体效果不出现衰退。

:::color5
**<font style="color:#601bde;">2. 自研的附加价值与个性化探索</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

除了上述严谨的技术与商业考量，自研预训练模型还具备以下附加价值：

+ **提升企业与个人技术影响力：** 成功构建自研模型能够有效展示企业的科研实力与技术深度，同时为参与研发的个人带来极大的职业成就感。
+ **支持个性化知识与价值观的注入：** 开发者可以在预训练语料中植入特定的知识彩蛋或定制化价值观。例如，通过特定频率（如每隔 100B token）向模型输入特定的表白话语，待模型上线后作为趣味互动（注：此操作需评估合规与职业风险，本文仅作技术探讨）。

:::color5
**<font style="color:#601bde;">1. 数据获取与爬取策略</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

启动大模型预训练的首要任务是筹备初始训练语料（通常建议在 10T 规模左右）。在模型训练启动后，亦可采用边训练边收集的动态策略。

+ **获取渠道与团队分工：** 数据获取途径包括网页爬取、电商平台采集及商业数据采购等。鉴于反爬虫机制（如 IP 封禁）及潜在的法律合规风险，强烈建议由专业的数据团队主导此类工作，而非由算法工程师兼任。
+ **高质量文档（如 PDF）解析：** 论文、书籍等高价值数据多为 PDF 格式。常规 Python 库在处理包含公式、表格的复杂 PDF 时效果欠佳。若采用 GPT-4 等先进大模型进行解析，成本往往过于高昂。推荐调用专业的 PDF 解析服务；若具备高质量的 PDF-Text 对齐数据，自研 OCR 模型亦是可行的备选方案。
+ **开源数据集的利用与局限：** 当前开源社区提供了丰富的数据集（如 FineWeb、The Pile、SkyPile、RedPajama），可作为冷启动的优质资源。然而，需清醒认识到，开源的中文大模型数据集往往并非发布方最核心、最纯净的数据，普遍存在一定程度的质量瑕疵。
+ **数据下载与大数据处理挑战：** 获取 Hugging Face 等平台的数据并非易事。常面临服务器外网限制（需切换至 `hf_mirror`）、下载速度缓慢（TB 级数据需耗时数日，且需手动分片、多进程多节点并行下载）以及文件数量庞大导致的文件系统卡死（如 `ls` 命令失效）等问题。这要求团队必须具备成熟的大数据集群处理能力。
+ **知识密度的认知与合成数据趋势：** 需深刻理解数据“知识密度”的差异（例如，《唐诗三百首》的知识密度远高于数百篇普通新闻）。高知识密度数据通常具有较高的获取成本。当前行业的新兴趋势是利用模型生成“合成高知识密度数据”，例如将冗长的新闻提炼为精简摘要后再输入模型，此举可大幅提升训练效率。
+ **组建专业数据团队的必要性：** 综上所述，若致力于开展严谨的预训练工作，组建专门的数据团队、实施专业爬虫或进行商业采购是必不可少的环节。仅依赖公开数据集进行简单清洗，难以满足高质量模型的训练需求。

:::color5
**<font style="color:#601bde;">2. 数据清洗与质量控制</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

数据清洗是整个数据处理流程中最为核心的环节，直接决定了模型的认知上限。

+ **引入模型打分机制：** 利用模型对预训练数据进行质量评分已成为行业标配（如 LLaMA 3、Qwen 2 技术报告所述）。业界普遍认为，在同等参数规模下，BERT 架构的表征能力优于 Transformer-Decoder 架构。因此，推荐选用 BERT 家族模型作为打分器，以兼顾效果与推理速度。训练数据可通过 GPT-4 标注或基于数据源质量规则（如设定特定源为高/低质量）进行构建。
    - **特殊格式数据的保护：** 值得注意的是，打分器通常会对 Code、Markdown、LaTeX 等格式化数据给出较低评分。必须在清洗前将这些数据隔离保护，避免被误删（例如，FineWeb 数据集就曾因清洗过度导致代码数据严重流失）。
    - **打分器训练的务实策略：** 训练打分器应保持务实心态，不应过度追求 100% 的准确率，达到可用标准即可。若耗费数月优化打分器，反而得不偿失。此外，打分器的上下文长度（如 4K）通常足以应对大部分场景，无需强求与长文本语料（如 32K）完全对齐。
+ **启发式规则的深度应用：** 打分器的结果仅是评估特征之一，规则过滤同样是数据清洗的“大杀器”。
    - **常见规则指标：** 包括文本长度阈值、特定 Token 比例、中英文占比、数字占比、是否包含特定字段（如 "http"）、以及是否包含低质量、敏感或违规关键词等。使用启发式规则进行过滤是高效且专业的做法。
    - **防止数据分布偏移：** 在应用规则时，必须警惕引入分布偏差。例如，若因“包含网址的数据质量低且英文占比高”而剔除所有高英文占比数据，可能导致模型退化为单语模型。因此，需频繁抽查（如使用 `vim` 查看）被过滤的数据样本，确保规则的合理性。
+ **数据脱敏与合规处理：** 数据脱敏是规避法律与隐私风险的必修课。需通过正则表达式等手段，严格剔除训练语料中的个人隐私信息（如人名、电话、邮箱）。此外，清除“转载自…”等冗余声明、不良信息及参考文献（References）等，亦属于广义的数据脱敏范畴。

:::color5
**<font style="color:#601bde;">3. 数据去重工程</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

对 TB 级别数据进行去重，是极具挑战性的工程任务。

+ **去重的必要性：** 互联网数据（如 Common Crawl）存在严重的重复引用与转载现象（如 URL 循环引用、多平台重复发布）。若不去重，模型将反复学习冗余信息，严重影响训练效率与泛化能力。
+ **去重粒度的权衡：** 理想状态下应进行句子级（Sentence-level）去重，但这将导致计算量呈指数级增长。建议根据团队算力与工程能力，在文档级（Document-level）与句子级去重之间量力而行。
+ **技术实现方案：** 必须依托大数据处理集群（如 Hadoop、Spark 等 MapReduce 框架）进行处理，切忌尝试使用 Python 循环等低效方式。核心算法通常采用 MinHash，其实现相对成熟。
+ **基于数据需求的动态阈值设定：** 去重工作需结合目标训练数据量来设定相似度阈值。例如，若目标为 10T 数据，可设定 80% 的相似度阈值；若目标为 5T，则可提高至 90%。目前尚无定论表明单条数据在预训练中学习多少次为最优，因此可大胆按需去重。对于因去重粒度较小而重复出现的文档，可通过在训练序列中拉开其物理距离（Token 间隔）来降低负面影响。

:::color5
**<font style="color:#601bde;">4. 数据分类与科学配比</font>**

:::

为实现精细化清洗与去重，需首先对数据进行分类。

+ **构建数据分类器：** 推荐训练一个基于 BERT 架构的分类器，将文档划分为新闻、百科、代码、Markdown 等宏观类目。
+ **差异化处理策略：**
    - **清洗阶段：** 针对“代码”与“知识类文本”，应设定不同的高质量判定阈值。
    - **去重阶段：** 不同类别数据的容忍度应有所区分（例如，“新闻”类相似度达 70% 即可丢弃，而“知识”类可放宽至 85%）。在剔除重复文档时，优先保留打分器评分较高的数据。
+ **主流数据配比参考：** 业界通常将数据划分为“知识（中/英）”、“代码”与“逻辑（Math/CoT）”三大类。对于中文大模型，常见的基础配比约为 `中文 : 英文 : 代码 = 4 : 4 : 2`（逻辑数据按实际收集量动态添加）。
+ **重视英文语料的比例：** 建议根据实际情况调整配比，但务必保持较高的英文语料占比。业界共识表明，当前中文开源语料在纯净度与数量级上仍逊色于英文语料，且中文的语言空间复杂度更高，学习难度更大。

:::color5
**<font style="color:#601bde;">5. 课程学习与数据顺序</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

预训练本质上是知识灌输的过程，知识的呈现顺序（即“课程学习”理念）对模型最终能力有显著影响。

+ **课程学习的核心理念：** 无论是“先易后难”还是“先净后脏”，课程学习均旨在证明：相同规模的训练数据，通过优化输入顺序，可产出能力差异巨大的模型。此理念已被广泛验证，成为预训练的标配策略。
+ **缓解灾难性遗忘：** 尽管 Next-Token 预测机制能确保模型拟合数据，但灾难性遗忘始终存在。合理的学习顺序（如将高频简单知识置于前期，低频困难知识合理穿插）有助于降低关键知识的遗忘率。
+ **上下文拼接策略（In-context Pretrain）：** 推荐参考 LLaMA 的实践（详见论文 [arXiv:2310.10638](https://arxiv.org/pdf/2310.10638)）：利用语义相似度，优先拼接相关性高的文档，构建语义连贯的上下文。
+ **Attention Mask 的取舍：** LLaMA 团队强调在拼接不相关文档时使用 Attention Mask（如 `sentenceA + "<sep>" + sentenceB` 且 B 看不到 A）效果显著。然而在实际工程中，多数团队发现是否使用 Mask 差异不大。此外，预训练阶段需培养模型在不同 Topic 间切换的能力（符合实际应用中单窗口多话题的交互场景）。因此，是否引入 Attention Mask 需通过实验严谨论证。

:::color5
**<font style="color:#601bde;">6. 高效数据流水线构建</font>**

:::

预训练模型需动态加载数据，而非一次性将 TB 级数据载入内存。

+ **进程解耦与预处理：** 预训练应包含独立的“数据处理进程”与“模型训练进程”。为避免 GPU 算力浪费，Tokenization、Concatenation 及 Padding 等操作必须在数据处理进程中提前完成。模型训练进程应直接获取处理好的 `token_id` 序列。
+ **数据复用与采样衰减：** 高质量数据可进行多轮训练（Epoch > 1）。数据处理进程在生成分片文件（如 `part-00000.jsonl`）时，应记录原始文档的使用次数，并相应降低高频数据的后续被选中概率。
+ **数据块粒度与容灾机制：** 鉴于训练过程中常遭遇硬件故障（烧卡）、Loss 爆炸等不可控因素，频繁回退至上一个 Checkpoint 是常态。建议将数据块大小设定为以 Billion (B) 为单位（如 1B、2B、4B），以减少版本回退带来的算力损失。
+ **动态 Checkpoint 保存技巧：** 为应对 Warmup 阶段动态数据块大小可能导致的保存逻辑失效问题，可引入“动态保存机制”：在训练脚本中增加轮询逻辑，一旦检测到特定目录下存在名为 `save` 的触发文件，即刻执行 `save_checkpoint` 操作。

:::color5
**<font style="color:#601bde;">7. 严谨的数据实验与 Scaling Law</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

在正式启动大规模训练前，必须在小规模模型上进行充分的实验验证，深刻理解 Scaling Law 的指导意义。

+ **实验阶段划分：**
    - **基础验证：** 在小模型（如 500B Token 训练量）上测试不同的数据配比与顺序，选取 Loss 曲线最平滑、下降最显著的方案（此阶段刷 Benchmark 意义有限）。
    - **参数推算：** 训练多个尺寸的小模型，结合 [Scaling Law](https://zhida.zhihu.com/search?content_id=247764221&content_type=Article&match_order=1&q=scaling_law&zhida_source=entity) 公式，推算目标大模型的最优数据配比、学习率及训练 Token 量。
    - **前沿探索：** 参考 LLaMA 与 DeepSeek 的技术报告，绘制 Loss 与 Benchmark 得分之间的 Scaling Law 曲线，以预判模型在特定训练量下的能力表现。
+ **遵循客观规律：** Scaling Law 虽发展放缓，但依然是指导训练的核心法则。脱离理论指导的盲目训练无异于随机试错。预训练成本高昂，实验阶段务必保持高度的鲁棒性与严谨性。

:::color5
**<font style="color:#601bde;">1. Tokenizer 的构建与优化</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

Tokenizer 的质量直接决定了模型的底层表征能力，是一项至关重要的“地基”工作。后期通过扩充词表来弥补初期设计的缺陷，往往治标不治本，且难以达到原生训练的惊艳效果。

+ **词表冲突风险：** 扩充词表时需警惕字典树逻辑导致的冲突。例如，若新增 Token“中华人民”并引入相应的 Merge 逻辑，可能导致原有 Token“中华人民共和国”永远无法被正确 Encode，进而造成相关知识的丢失。
+ **训练环境与算法：** 建议在具备大内存的 CPU 机器上，使用海量通用数据集，基于 BPE 或 BBPE 算法进行训练。
+ **关键细节把控：**
    1. **数字切分：** 务必进行数字切分，以避免模型在处理类似“9.9 与 9.11 大小比较”等逻辑问题时出现偏差。
    2. **压缩率控制：** 需平衡解码效率与知识容量。压缩率过低（字多词少）影响推理速度；压缩率过高（词汇过度组合）则削弱模型的知识泛化能力。通常，优秀中文大模型的压缩率约为 1 个 Token 对应 1.5 个汉字。
    3. **数据净化：** 手动剔除词表中的脏数据（如色情、赌博等违规词汇），避免重蹈部分开源模型词表泄露暴露出不良 Token 的覆辙。
    4. **垂直领域适配：** 若明确业务场景（如医疗），应提前将专业术语（如“阿莫西林”）作为独立 Token 纳入词表，以提升特定领域的压缩率与表达准确性。
    5. **多语言覆盖：** 确保中英文的高覆盖率，小语种的支持则视具体业务需求而定。
    6. **预留 Buffer 空间：** Tokenizer 的 `vocab_size` 应比模型的 `embedding_size` 预留约 1000 个位置的 Buffer，以便在后续 Alignment 阶段引入必要的特殊控制 Token。

:::color5
**<font style="color:#601bde;">2. 模型架构选型原则</font>**

:::

预训练成本极高，架构选型应坚持“稳健至上，谨慎创新”的原则。

+ **拥抱主流架构：** 强烈建议采用业界验证成熟的 LLaMA 架构（RoPE + GQA + RMSNorm + SwiGLU）。减少非必要的微观结构创新，即可大幅降低踩坑概率。
+ **参数共享策略：** 对于极小规模（如 1B 左右）的模型，建议共享 Embedding 与 LM_Head 的参数，以提升隐藏层参数在全局中的占比；对于较大规模模型则无此必要。
+ **警惕伪创新风险：** 除非具备极其坚实的理论基础（如严密的数学推导）且经过长期、大规模的鲁棒性实验验证，否则不建议仅凭小模型上的短期实验结果（如 Loss 下降较快）就贸然更改底层架构。盲目创新可能导致在投入巨额算力后暴露出致命缺陷，带来不可挽回的沉没成本与职业风险。

:::color5
**<font style="color:#601bde;">3. 模型参数规模与超参数设定</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

模型规模的设定需综合考量训练资源与实际部署成本。

+ **模型规模（Size）的评估维度：**
    - **训练算力约束：** 根据团队拥有的 GPU 资源、计划训练周期、可用数据量以及框架吞吐量，逆向推算可支撑的模型规模。建议尽量对齐大厂的主流尺寸，以便于效果对标并减少未知风险。
    - **推理算力约束：** 需深刻理解部署硬件的显存极限。例如，33B 模型通常逼近单张 80G 显卡的推理极限，而 72B 模型则逼近双卡极限。若训练一个 40B 模型，其推理成本与 72B 模型处于同一量级（均需双卡），则显得不够经济。因此，模型尺寸的设计（如 65B、70B、72B 等）需与目标推理硬件的显存规格紧密契合。
+ **超参数的“匀称性”与硬件友好性：**
    - 优秀的模型架构应保持横向（Hidden Size）与纵向（Layer Number）的合理比例。
    - **硬件对齐要求：** 为适配分布式训练框架的并行策略，超参数应尽量设定为 2、4、8、64、128 等数值的整数倍。
        * `layer_num`：应具备尽可能多的质因数，以完美支持流水线并行（Pipeline Parallelism，需满足 `layer_num % pipeline_size == 0`）。
        * `num_head`：建议为 8 的整数倍，以优化张量并行（Tensor Parallelism）。张量并行通常在单机 8 卡内效率最高，跨机通讯会显著降低性能。
        * `hidden_states` 与 `vocab_size`：建议设定为 128 的整数倍，以充分利用底层算子加速。
+ **序列长度（Seq_Len）的演进策略：** 初始训练阶段切勿直接采用超长文本（如 32K/200K），以免 Attention 计算量（呈平方级增长）压垮算力。推荐采用 RoPE 的 NTK 外推方案：前期（约 90% 数据量）使用 4K/8K 长度与较小的 RoPE Base；后期（约 10% 数据量）切换至 32K/64K 长度与较大的 RoPE Base 进行长文本适应性训练。

:::color5
**<font style="color:#601bde;">4. 分布式训练框架对比：Megatron vs DeepSpeed</font>**

:::

对于从零开始的 TB 级预训练，强烈推荐使用 Megatron；对于 Billion 级别的 Continue-Pretrain，DeepSpeed 亦是可行选择。

+ **Megatron 的优势：**
    1. **极致的训练效率：** 其张量并行与流水线并行优化极佳，RoPE 等关键操作已封装为高效的 Apex 算子，整体吞吐量远超常规实现。
    2. **细粒度的参数控制：** 提供极其丰富的配置项（如 `argument.py`），允许开发者对各类网络层细节进行微操。
    3. **极速的模型加载：** 千亿参数模型的加载时间通常控制在分钟级，极大提升了 Debug 效率。
+ **Megatron 的劣势：**
    1. **陡峭的学习曲线：** 对新手极不友好，要求深入理解 PyTorch 多机通讯机制，且与 Hugging Face 格式的转换脚本维护成本较高。
    2. **官方代码库稳定性欠佳：** 官方仓库常伴随 Bug，需预留充足的调试时间，盲目追新（频繁 `git pull`）可能导致工作量剧增。
+ **DeepSpeed 的优势：**
    1. **极简的代码侵入性：** 接口设计高度抽象（如 `model.trainer`），上手极快。
    2. **庞大的社区生态：** 在 Alignment 阶段拥有极高的市场占有率，开源参考代码丰富。
+ **DeepSpeed 的劣势：**
    1. **算力损耗：** 相比 Megatron，可能存在 10%~30% 的训练效率折损（尽管社区已补充了复杂的并行策略，但牺牲了易用性）。
    2. **令人崩溃的加载速度：** 模型加载极其缓慢，大模型启动往往耗时半小时以上，严重制约 Debug 效率。
    3. **微操受限：** 高度封装导致底层逻辑修改困难。
    4. **隐蔽的 Bug 机制：** 遇到异常现象时，往往难以排查是自身代码问题还是框架底层缺陷。

_(注：无论采用何种框架，务必确保开启 Flash Attention 以优化注意力机制计算。)_

:::color5
**<font style="color:#601bde;">5. 训练效率优化与 Loss 监控</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

+ **通讯开销优化原则：** 牢记通讯速度法则：`卡内通讯 > 卡间通讯 > 机间通讯`。在显存充裕的前提下，尽量减少张量并行、流水线并行及序列并行（Sequence Parallelism）的引入；尽量避免开启 Offload（显存与内存间的数据搬运极其耗时）与重算（Activation Checkpointing）。数据并行（Data Parallelism）是唯一不显著牺牲算力的并行方式。
+ **Loss 曲线的深度分析：** TensorBoard 中的 Loss 曲线是预训练过程最重要的健康指标。
    1. **细分 Channel Loss：** 必须分别监控中文知识、英文知识、代码等不同数据类别的 Loss 走向。
    2. **严防 Loss Spike：** 无论 Loss 突然飙升还是骤降，均需高度警惕，这通常是数据质量问题（如乱码导致飙升，大量换行符导致骤降）的信号。若确认数据异常，应果断回退至上一个健康的 Checkpoint。
    3. **优化器参数调优：** 若数据无异常仍出现 Spike，需检查梯度（Grad）状态，并参考权威技术报告调整 AdamW 优化器的超参数。通常熬过训练初期的波动，待 Loss 平稳降至 2~3 左右后，训练过程将趋于稳定。

:::color5
**<font style="color:#601bde;">6. 标准化训练流程</font>**

:::

当数据与代码准备就绪后，预训练通常遵循以下四个阶段的标准化流程：

1. **Warmup（预热期）：** 学习率从零缓慢爬升至设定的最大值。
2. **主训练期：** 维持较高学习率，采用 Cosine、Constant 或相应的 Decay 策略（具体策略需参考技术报告或小模型实验结论）。
3. **长文本适应期：** 调整 RoPE Base 与 Seq_Len，引导模型适应长上下文。
4. **Anneal（退火收尾期）：** 引入高质量精选数据或 IFT（Instruction Fine-Tuning）数据，强化模型的特定能力，为后续的 Benchmark 评估做准备。

预训练启动后，若无硬件故障或 Loss 异常，通常无需过多人工干预，只需做好监控与必要的 Checkpoint 回退即可。

# **评估篇**
:::color3
**简介：**预训练阶段的评估相对纯粹，主要聚焦于模型对基础知识的掌握程度，无需过度考察指令遵循或安全对齐能力。本章介绍了 PPL 监控、Benchmark 的创新应用及概率探针等实用评估手段。

:::

:::color5
**<font style="color:#601bde;">1. PPL (Perplexity) 监控</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

观察测试集上的 Loss（PPL）是衡量模型效果的基础手段。

+ **构建专属测试集：** 准备包含百科、逻辑、代码等高质量语料的独立测试集（确保未在训练集中泄露）。
+ **趋势分析：** 持续监控模型在该测试集上的 Loss 表现。合理的趋势应是持续下降并最终趋于平稳，且与训练集 Loss 曲线保持高度一致。
+ **横向对比的局限性：** 需注意，由于不同模型的 Tokenizer 压缩率存在差异，直接对比不同模型的绝对 Loss 值缺乏科学依据（例如，压缩率极低、偏向字粒度的 Tokenizer 天生具有更低的 Loss）。但作为经验参考，优秀模型在通用知识测试集上的 Loss 通常应降至 2 以下。

:::color5
**<font style="color:#601bde;">2. Benchmark 的创新应用与防作弊策略</font>**

:::

在评估开源模型或接手他人 Checkpoint 时，必须警惕 Benchmark 刷榜现象（即训练数据中混入测试集）。

+ **传统 Benchmark 的局限：** 当前主流 Benchmark 多为缺乏 CoT（思维链）过程的纯选择题，难以全面、客观地衡量模型的真实推理能力。
+ **生成式改造策略：** 为规避模型“死记硬背”选项答案，建议对 Benchmark 进行花式改造，采用生成式方法进行评估。例如：
    - 将选项内容直接拼接入 Prompt，要求模型结合文本生成答案。
    - 篡改正确选项的标识（如将 B 选项内容改为“其他答案全错”），测试模型的抗干扰能力。
    - 替换选项序号（如将 A/B/C/D 替换为 一/二/三/四）。
    - 将多选题降维为单选题。
    - 采用先提供知识再提问的 Few-shot 形式，引导模型输出。
+ **格式控制与指标选取：** 若担心模型输出格式不可控，可利用 Few-shot 引导，或通过限制 `max_new_token` 及设置 `StoppingCriteria`（如遇换行符即停止）来截断输出。评估指标务必采用准确率（ACC），避免使用 BLEU 或 ROUGE 等难以进行 Case 分析的抽象指标。核心关注点在于模型得分是否随训练过程持续提升，而非绝对分值的高低。

:::color5
**<font style="color:#601bde;">3. 概率探针 (Probability Probes) 技术</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

概率探针是一种从底层概率分布视角监控模型特定知识掌握程度与遗忘情况的有效方法。

+ **核心思路：** 持续追踪特定 Token 或句子的生成概率变化趋势。探针测试集通常需要人工精细构造。
+ **应用示例：**
    1. **事实知识追踪：** 监控 `Prob('北京' | '中国的首都是')` 的概率值是否随训练推进而稳步上升。
    2. **价值观与立场验证：** 监控特定陈述句（如符合正确价值观的表述）的整体 PPL 是否持续下降，而对立表述的 PPL 是否上升。
    3. **对比探针：** 验证特定正向表述的概率是否始终大于负向表述。
    4. **基础格式遵循：** 监控 `Prob('{' | '请以 JSON 格式输出')`，评估模型对结构化输出的敏感度。
+ **趋势重于绝对值：** 与 Benchmark 改造类似，概率探针的设计高度灵活，评估的核心在于观察概率指标的动态变化趋势，以此诊断模型的健康状态。

**<font style="color:#601bde;">1. 结语与核心观点重申</font>**

大模型预训练的全链路工程大抵如上所述，每一个环节均对最终模型的质量起着举足轻重的作用。

业界曾有观点认为“数据清洗是缺乏技术含量的脏活累活”，对此本文持坚决反对态度。在底层 Infra 团队已保障分布式训练框架（如 Megatron）稳定运行的前提下，单纯的模型训练执行（如修改参数、启动脚本、处理故障重启）反而成为了相对标准化的常规操作。相反，在海量数据清洗与配比过程中展现出的工程直觉与策略创新，往往能够为模型效果带来质的飞跃。这也是本文在“数据篇”倾注最多笔墨的根本原因。预训练不仅是算力的比拼，更是对数据深刻理解与精细化工程能力的全面考验。
