# Agentic WorkFlow<font style="color:#D22D8D;">（by草莓师姐）</font>
:::color3
**简介：****<font style="color:rgb(25, 27, 31);">工作流（Workflow）</font>**<font style="color:rgb(25, 27, 31);"> 指的是一系列</font>**<font style="color:rgb(25, 27, 31);">相互关联的步骤</font>**<font style="color:rgb(25, 27, 31);">，旨在完成特定任务或目标。最简单的工作流是</font>**<font style="color:rgb(25, 27, 31);">确定性的（Deterministic）</font>**<font style="color:rgb(25, 27, 31);">，即它们遵循</font>**<font style="color:#ED740C;">预定义的步骤序列，无法适应新信息或变化的环境</font>**<font style="color:rgb(25, 27, 31);">。例如，一个</font>**<font style="color:rgb(25, 27, 31);">自动化的报销审批工作流</font>**<font style="color:rgb(25, 27, 31);">可能如下所示：如果费用标签为“餐饮”，且金额小于$30，则自动批准。</font>

<font style="color:rgb(25, 27, 31);">一些工作流利用大型语言模型（LLMs）或其他机器学习模型来增强其能力。这些通常被称为 </font>**<font style="color:rgb(25, 27, 31);">AI 工作流</font>**<font style="color:rgb(25, 27, 31);">，可分为</font>**<font style="color:rgb(25, 27, 31);">代理型（Agentic）</font>**<font style="color:rgb(25, 27, 31);">和</font>**<font style="color:rgb(25, 27, 31);">非代理型（Non-Agentic）</font>**<font style="color:rgb(25, 27, 31);">两种：</font>

+ **<font style="color:rgb(25, 27, 31);">非代理型 AI 工作流</font>**<font style="color:rgb(25, 27, 31);">：LLM 根据输入的指令生成输出。例如，</font>**<font style="color:rgb(25, 27, 31);">文本摘要</font>**<font style="color:rgb(25, 27, 31);">工作流的流程可能是：接收长文本 → 让 LLM 进行总结 → 输出摘要。这种流程仅仅依赖 LLM 的文本处理能力，并不具备自主决策或任务执行能力，因此不属于代理型工作流。</font>
+ **<font style="color:rgb(25, 27, 31);">代理型 AI 工作流</font>**<font style="color:rgb(25, 27, 31);">：由一个或多个</font>**<font style="color:rgb(25, 27, 31);">AI 代理（Agents）动态执行</font>**<font style="color:rgb(25, 27, 31);">一系列步骤，以完成特定任务。代理在用户授予的权限范围内，具备一定程度的</font>**<font style="color:rgb(25, 27, 31);">自主性</font>**<font style="color:rgb(25, 27, 31);">，可以收集数据、执行任务，并做出实际决策。此外，代理型工作流利用 AI 代理的</font>**<font style="color:rgb(25, 27, 31);">推理能力、工具使用能力和持久记忆能力</font>**<font style="color:rgb(25, 27, 31);">，使传统工作流</font>**<font style="color:rgb(25, 27, 31);">更具响应性、适应性和自我进化能力</font>**<font style="color:rgb(25, 27, 31);">。</font><font style="color:#D22D8D;">（by草莓师姐）</font>

**<font style="color:#601BDE;">3.局限性</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

<font style="color:rgb(25, 27, 31);">尽管具有诸多优势和创新功能，AI代理也存在一些挑战和局限性。由于其概率性质，AI代理本质上为工作流增添了复杂性。仅仅因为代理可以用于自动化流程，并不意味着它们应该被使用。以下是Agentic工作流的一些显著挑战和局限性：</font>

+ **<font style="color:rgb(25, 27, 31);">对于简单任务的过度复杂性</font>**<font style="color:rgb(25, 27, 31);">。在用于简单工作流（如表单输入或基本数据提取）时，AI代理可能会增加额外的开销。在确定性、基于规则的自动化足够的情况下，引入代理可能会导致低效、额外费用，甚至可能降低性能。</font>
+ **<font style="color:rgb(25, 27, 31);">随着自主性增加导致可靠性降低</font>**<font style="color:rgb(25, 27, 31);">。随着代理在工作流中获得更多的决策权，其概率性质可能会引入不可预测性，使得输出不太可靠且更难控制。实施并持续维护代理的防护措施，并定期审查其授权权限至关重要。</font>
+ **<font style="color:rgb(25, 27, 31);">伦理和实践考量</font>**<font style="color:rgb(25, 27, 31);">。并非所有决策都应委托给AI系统。在高风险或敏感领域使用代理时，需要谨慎监督，以确保负责任的部署并防止意外后果。</font>

<font style="color:rgb(25, 27, 31);">鉴于这些局限性，建议在使用代理之前，花时间反思其是否在特定工作流中确实必要。以下问题可以帮助您做出判断：</font>

+ <font style="color:rgb(25, 27, 31);">任务是否足够复杂，需要适应性决策，还是确定性方法足够？</font>
+ <font style="color:rgb(25, 27, 31);">简单的AI辅助工具（如没有代理的RAG）能否达到相同的效果？</font>
+ <font style="color:rgb(25, 27, 31);">工作流是否涉及不确定性、变化条件或多步推理，代理能更有效地处理这些问题吗？</font>
+ <font style="color:rgb(25, 27, 31);">给予代理自主性带来的风险是什么，能否有效降低这些风险？</font>

  


# Agent
## Agent介绍<font style="color:#D22D8D;">（by草莓师姐）</font>
:::success
**背景：**随着大语言模型的横空出世，生成式AI应用也在不断发展，下图列出了生成式AI应用分级，共分为5级：

+ L1 Tool，人类完成所有工作，基本没有任何显性的AI辅助，只是比较简单的工具，会被逐渐升级淘汰；
+ L2 Chatbot，人类直接完成绝大部分工作，人类向AI询问意见，了解信息，AI提供信息和建议但不直接处理工作；
+ L3 Copilot（副驾驶），人类和AI进行协作，工作量相当，AI根据人类要求完成工作初稿，人类进行目标设定，修改调整，最后确认；
+ **L4 Agent**，AI完成绝大部分工作，人类负责设定目标、提供资源和监督结果，AI完成任务拆分，工具选择，进度控制，实现目标后自主结束工作；
+ L5 Intelligence，完全无需人类监督，AI自主拆解目标，寻找资源，选择并使用工具，完成全部工作，人类只需给出初始目标。

当发展到L4级别时，Agent（即智能体）的能力已经越来越接近通用人工智能，可以借助AI完成绝大部分的工作。相对来看，L3和L4级别的AI应用价值更大，但需要大语言模型的能力来进一步支持升级。

**<font style="color:#601BDE;">0.</font>****<font style="color:#601BDE;">AI Agent是什么？</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

**<font style="color:#74B602;">AI Agent</font>**<font style="color:rgba(0, 0, 0, 0.9);">又是什么呢？中文翻译过来是“</font>**<font style="color:#74B602;">人工智能代理</font>**<font style="color:rgba(0, 0, 0, 0.9);">”，即通过LLM为大脑，响应需求，进行意图理解，规划拆解任务，调用工具，执行动作并生成内容。</font>

<font style="color:rgba(0, 0, 0, 0.9);">AI Agent可以把Function call理解为其实现目标的“</font>**<font style="color:#74B602;">动作单元</font>**<font style="color:rgba(0, 0, 0, 0.9);">”，就是那个可以调用的工具。</font>

<font style="color:rgba(0, 0, 0, 0.9);">比如，你想睡觉了，那就只要设计一个AI Agent，说一句话：“现在我要睡觉了”。 LLM就开始进行意图识别，自己拆解任务，用户要睡觉，意味着要干三件事，关窗帘、关灯、空调调到睡眠模式。然后调用这三个电器的Function call，执行任务。</font>

<font style="color:rgba(0, 0, 0, 0.9);">AI Agent就像你的</font>**<font style="color:#74B602;">管家机器人</font>**<font style="color:rgba(0, 0, 0, 0.9);">一样，你布置任务，它代你执行，就像你的代理人一样。所以你现在理解了这里面的“Agent”含义了吧。</font>

:::color5
**<font style="color:#601BDE;">1.任务示例</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

1. **流程示例**：
    - <font style="color:rgb(51, 51, 51);">用户问：“今天北京的天气如何？”</font>
    - **<font style="color:rgb(51, 51, 51);">Function Call</font>**<font style="color:rgb(51, 51, 51);">：解析意图，提取关键词“北京”“天气”。</font>
    - **<font style="color:rgb(51, 51, 51);">Tool Call</font>**<font style="color:rgb(51, 51, 51);">：调用天气API获取数据。</font>
    - **<font style="color:rgb(51, 51, 51);">Function Call</font>**<font style="color:rgb(51, 51, 51);">：将API返回的JSON数据转换为自然语言回复。</font>
    - <font style="color:rgb(51, 51, 51);">输出：“北京今天晴，气温25℃。”</font>
2. **框架支持**：  
某些框架（如LangChain、AutoGPT）将两者统一抽象为“工具”，Agent通过决策模块（如LLM）选择调用内部函数或外部工具，例如：

```python
# LangChain示例：定义工具（可能是函数或API）
tools = [
    Tool(name="internal_parse", func=parse_intent, description="解析用户意图"),
    Tool(name="search_web", func=google_api_call, description="调用搜索引擎")
]
agent.run("用户问题", tools=tools)
```

## ReAct & <font style="color:rgb(1, 1, 1);">Plan-and-Execute 对比</font><font style="color:#D22D8D;">（by草莓师姐）</font>
:::color3
**<font style="color:rgb(51, 51, 51);">简介</font>**<font style="color:rgb(51, 51, 51);">：</font><font style="color:rgb(25, 27, 31);">目前单一 Agent 的实现方案主要有以下两种：</font>

**<font style="color:rgb(51, 51, 51);">ReAct：</font>**<font style="color:rgb(51, 51, 51);">通过</font>**<font style="color:rgb(51, 51, 51);">动态交替推理与行动</font>**<font style="color:rgb(51, 51, 51);">实现目标，强调实时交互式的问题解决。其工作流呈现"感知-推理-行动-反馈"的循环结构，类似人类面对未知问题时的试错过程。</font>

**<font style="color:rgb(1, 1, 1);">Plan-and-Execute：</font>**<font style="color:rgb(51, 51, 51);">采用</font>**<font style="color:rgb(51, 51, 51);">先规划后执行</font>**<font style="color:rgb(51, 51, 51);">的分阶段策略，强调结构化的问题分解。其核心流程为"目标分解→生成计划→按序执行"，类似项目管理中的甘特图制定。</font>

:::

:::color5
**<font style="color:#601BDE;">1.实现逻辑对比</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

:::success
**<font style="color:rgb(1, 1, 1);">ReAct（P人）</font>**

<font style="color:rgb(1, 1, 1);">将任务分解为思考(Reasoning)和行动(Acting)两个交替进行的阶段。</font>

1. <font style="color:rgb(1, 1, 1);">通过少样本Prompt引导大模型生成思考链（Thought Chain）确定下一步行动</font>
2. <font style="color:rgb(1, 1, 1);">执行工具调用（Action）</font>
3. <font style="color:rgb(1, 1, 1);">观察结果并迭代优化。</font>

:::

:::success
**<font style="color:rgb(1, 1, 1);">Plan-and-Execute（J人）</font>**

<font style="color:rgb(1, 1, 1);">将任务分为规划（Planning）和执行（Execution）两个明确的阶段。</font>

<font style="color:rgb(1, 1, 1);">1， 首先生成完整的任务执行计划（通常是DAG图）</font>

2. <font style="color:rgb(1, 1, 1);">然后按计划批量执行子任务。</font>

:::

**ReAct工作机制**

+ **<font style="color:rgb(51, 51, 51);">推理阶段</font>**<font style="color:rgb(51, 51, 51);">：分析当前状态，生成候选动作（如："用户需要天气信息，应调用API"）</font>
+ **<font style="color:rgb(51, 51, 51);">行动阶段</font>**<font style="color:rgb(51, 51, 51);">：执行动作并观察结果（如：调用天气API但返回错误）</font>
+ **<font style="color:rgb(51, 51, 51);">适应性调整</font>**<font style="color:rgb(51, 51, 51);">：根据反馈重新推理（如："API故障，改用备用数据源"）</font>

**Plan-and-Execute工作机制**

+ **<font style="color:rgb(51, 51, 51);">规划阶段</font>**<font style="color:rgb(51, 51, 51);">：深度任务分解（如：将"组织会议"拆解为预定场地、发送邀请等子任务）</font>
+ **<font style="color:rgb(51, 51, 51);">执行阶段</font>**<font style="color:rgb(51, 51, 51);">：严格按计划推进，通常仅在严重偏差时重新规划</font>
+ **<font style="color:rgb(51, 51, 51);">验证机制</font>**<font style="color:rgb(51, 51, 51);">：设置检查点确保阶段目标达成</font>

:::color5
**<font style="color:#601BDE;">2.优缺点对比</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

| | 优点 | 缺点 |
| --- | --- | --- |
| ReAct | 1. <font style="color:rgb(1, 1, 1);">决策过程透明可解释</font><br/>2. <font style="color:rgb(1, 1, 1);">工具调用灵活，可动态调整策略</font><br/>3. <font style="color:rgb(1, 1, 1);">适合需要复杂推理的任务</font> | 1. <font style="color:rgb(1, 1, 1);">每个步骤都需要</font>**<font style="color:#74B602;">LLM参与，token消耗大</font>**<br/>2. <font style="color:rgb(1, 1, 1);">容易在复杂任务中陷入循环</font><br/>3. <font style="color:rgb(1, 1, 1);">执行效率相对较低</font> |
| <font style="color:rgb(1, 1, 1);">Plan-and-Execute</font> | 1. <font style="color:rgb(1, 1, 1);">显著减少LLM调用次数，降低成本</font><br/>2. <font style="color:rgb(1, 1, 1);">任务执行更加高效</font><br/>3. <font style="color:rgb(1, 1, 1);">适合步骤明确的复杂任务</font> | 1. <font style="color:rgb(1, 1, 1);">对初始规划的准确性要求高</font><br/>2. <font style="color:rgb(1, 1, 1);">难以处理高度动态的任务</font><br/>3. <font style="color:rgb(1, 1, 1);">子任务间的依赖关系复杂时容易失败</font> |

**简介：**<font style="color:rgb(25, 27, 31);">随着行业对Agent能力的期待越来越高，MAS越来越多地被使用也是大势所趋。尽管目前MAS在解决实际问题的准确性确实还不如Workflow，但潜力和想像空间不是后者所能比拟的。随着技术进步和行业法规的完善，MAS必定可以展现出指数级强于ChatBot的能力。</font>

**参考：**<font style="color:rgb(25, 27, 31);"></font>[大模型领域MultiAgent与Agent调用多个tool有啥大的区别么？](https://www.zhihu.com/question/4233505972/answer/85042959959)

<font style="color:rgb(25, 27, 31);">从</font>**<font style="color:rgb(25, 27, 31);">功能、成本，设计模式</font>**<font style="color:rgb(25, 27, 31);">三个角度考虑</font>**<font style="color:rgb(25, 27, 31);">：</font>**

1. **<font style="color:rgb(25, 27, 31);">功能：</font>**<font style="color:rgb(25, 27, 31);">要达到一个复杂任务被出色完成的目标，最可靠的方式是</font>**<font style="color:#74B602;">让每个Agent只扮演特定岗位的角色，每个Agent也只完成足够简单的任务。</font>**
2. **成本：**<font style="color:rgb(25, 27, 31);">从管理的角度，一个团队中每个人需要各司其职、也需要人尽其用，很大一方面原因是需要</font>**<font style="color:rgb(25, 27, 31);">控制人力成本</font>**<font style="color:rgb(25, 27, 31);">。理想状态下，一个项目/任务经过合理拆解后，简单的部分交给junior/associate的岗位去做，复杂的部分交给senior/expert的岗位去做，这样可以有效地控制项目/任务的人力成本</font>
3. **<font style="color:rgb(25, 27, 31);">设计模式：</font>**<font style="color:rgb(25, 27, 31);">由于Agent模式不仅在可扩展性、可重用性等设计理念上代表了更先进生产力的发展要求，而且</font>**<font style="color:rgb(25, 27, 31);">在处理复杂任务时的性能确实强于</font>****<font style="color:#2F4BDA;">只面向过程的chatbot</font>****<font style="color:rgb(25, 27, 31);">，</font>**<font style="color:rgb(25, 27, 31);">也代表了广大用户的根本利益，所以Agent设计模式开始越来越多地被用户接受。</font>

**<font style="color:#601BDE;">1.Multi Agent & Agent 区别</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

`MultiAgents` 和 `Agent-tools` 模式的核心区别在于**角色分工**和**功能实现**：

+ **MultiAgent**: **<font style="color:#74B602;">多个LLM，角色分工明确（角色扮演很重要！）</font>**，偏向协作解决复杂任务，主要通过 自然语言交互完成协调（LLMs 最强大之处在于可以用人类语言进行交流）
+ **Agent-tools**: 单个Agent调用工具（通常为API或功能模块）完成**<font style="color:#74B602;">特定功能</font>**，偏向任务执行和效率。

**核心区别之处：**

+ `MultiAgents` 更强调**智能间协作，擅长**复杂推理和动态协调）
+ `Agent-tools` 更注重现成**功能调用**（如计算、检索）。

## <font style="color:rgb(25, 27, 31);">多智能体失败分析 Why Do Multi-Agent LLM Systems Fail</font>
:::success
**<font style="color:rgb(51, 51, 51);">背景</font>**<font style="color:rgb(51, 51, 51);">：</font><font style="color:rgb(25, 27, 31);">近年来，基于</font>[<font style="color:rgb(9, 64, 142);">大型语言模型</font>](https://zhida.zhihu.com/search?content_id=255767496&content_type=Article&match_order=1&q=%E5%A4%A7%E5%9E%8B%E8%AF%AD%E8%A8%80%E6%A8%A1%E5%9E%8B&zhida_source=entity)<font style="color:rgb(25, 27, 31);">（LLM）的智能体系统因其在复杂多步骤任务中的动态交互能力备受关注。多Agent系统（MAS）通过协作实现</font>**<font style="color:#74B602;">任务分解、并行处理、上下文隔离</font>**<font style="color:rgb(25, 27, 31);">等优势，被广泛应用于软件开发、药物发现、科学模拟等领域。然而，尽管MAS的潜力巨大，其性能在</font>**<font style="color:#117CEE;">主流基准测试中并未显著超越单Agent系统</font>**<font style="color:rgb(25, 27, 31);">，甚至低于简单的“best-of-N sampling”方法。例如，开源框架ChatDev在某些任务中的正确率仅为25%。这一现象引发核心问题：</font>**<font style="color:rgb(25, 27, 31);">为何多智能体系统频繁翻车？</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

![](https://cdn.nlark.com/yuque/0/2025/png/29769680/1744267738709-db9c4ef0-a235-41b2-ba14-b10ad8f284b4.png)

:::

:::color3
**简介：**<font style="color:rgb(25, 27, 31);">研究者采用</font>[**<font style="color:rgb(9, 64, 142);">Grounded Theory</font>**](https://zhida.zhihu.com/search?content_id=255767496&content_type=Article&match_order=1&q=Grounded+Theory&zhida_source=entity)<font style="color:rgb(25, 27, 31);">，系统分析了5个主流开源MAS框架（包括ChatDev、MetaGPT等）在150多个任务中的trace，涵盖超过15,000行对话文本。</font><font style="color:#D22D8D;">（by草莓师姐）</font>

**paper：**[**https://arxiv.org/pdf/2503.13657**](https://arxiv.org/pdf/2503.13657)

:::color5
**<font style="color:#601BDE;">1.MAS框架</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

![](https://cdn.nlark.com/yuque/0/2025/png/29769680/1744268332554-aa35bb78-37c4-4bb5-bc58-6ceb99246020.png)

<font style="color:rgb(25, 27, 31);">整体研究流程如下：</font>

+ **<font style="color:rgb(25, 27, 31);">数据采集</font>**<font style="color:rgb(25, 27, 31);">：通过理论抽样选择多样化的任务和MAS框架，确保覆盖不同领域（如软件开发、工具调用代理）。</font>
+ **<font style="color:rgb(25, 27, 31);">开放编码</font>**<font style="color:rgb(25, 27, 31);">：六位专家注释者标注对话轨迹，识别失败模式，并通过持续比较分析迭代生成初始分类。</font>
+ **<font style="color:rgb(25, 27, 31);">一致性验证</font>**<font style="color:rgb(25, 27, 31);">：三位专家独立标注15条轨迹，最终达成Cohen's Kappa一致性评分0.88，表明分类高度可靠。</font>

<font style="color:rgb(25, 27, 31);">最终，研究者提出</font>**<font style="color:rgb(25, 27, 31);">多智能体系统失败分类法（</font>**[**<font style="color:rgb(9, 64, 142);">MASFT</font>**](https://zhida.zhihu.com/search?content_id=255767496&content_type=Article&match_order=1&q=MASFT&zhida_source=entity)**<font style="color:rgb(25, 27, 31);">）</font>**<font style="color:rgb(25, 27, 31);">，涵盖3大类14种细粒度失败模式，并通过LLM自动化标注管道（Cohen's Kappa 0.77）实现高效评估。</font>

**<font style="color:#601BDE;">2.MAS失败分类</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

<font style="color:rgb(25, 27, 31);">将失败模式分为三大类别，具体如下：</font>

# Function Call / Tool Call / MCP
## 对比
:::color3
**<font style="color:rgb(51, 51, 51);">简介</font>**<font style="color:rgb(51, 51, 51);">：在Agent架构中，</font>**<font style="color:rgb(51, 51, 51);">Tool Call</font>**<font style="color:rgb(51, 51, 51);">、</font>**<font style="color:rgb(51, 51, 51);">Function Call、MCP</font>**<font style="color:rgb(51, 51, 51);">是三种关键的任务执行机制.</font>

+ **<font style="color:rgb(51, 51, 51);">Function Call</font>**<font style="color:rgb(51, 51, 51);">是Agent的“内功”，处理可控的内部逻辑。</font>
+ **<font style="color:rgb(51, 51, 51);">Tool Call</font>**<font style="color:rgb(51, 51, 51);">是Agent的“外功”，扩展能力边界，与外部世界交互。</font>
+ **<font style="color:rgb(51, 51, 51);">MCP</font>**<font style="color:rgb(51, 51, 51);">：</font><font style="color:rgb(25, 27, 31);">定义了应用程序和 AI 模型之间交换上下文信息的方式。这使得开发者能够</font>**<font style="color:rgb(25, 27, 31);">以一致的方式将各种数据源、工具和功能连接到 AI 模型</font>**<font style="color:rgb(25, 27, 31);">（一个中间协议层），就像</font>**<font style="color:#ED740C;"> USB-C</font>**<font style="color:rgb(25, 27, 31);"> 让不同设备能够通过相同的接口连接一样。MCP 的目标是创建一个通用标准，使 AI 应用程序的开发和集成变得更加简单和统一。</font>

<font style="color:rgb(51, 51, 51);">三者结合使Agent既能高效处理结构化任务，又能灵活应对动态、实时的外部需求。</font>

:::

**<font style="color:#601BDE;">0.MCP和Function Call的区别与联系</font>**

<font style="color:rgba(0, 0, 0, 0.9);">MCP并非Function Calling的替代品，而是通过标准化协议解决了碎片化集成的问题，同时为Function Calling提供了更广阔的协作空间。两者的关系类似于</font>**<font style="color:#74B602;">“高速公路与汽车”——MCP是连接万物的基础设施，Function Calling则是灵活高效的交通工具</font>**<font style="color:rgba(0, 0, 0, 0.9);">，二者结合才能实现AI应用的规模化落地。</font>

:::color5
**<font style="color:#601BDE;">1.Function Call（函数调用）</font>**

:::

+ **定义**：  
指Agent在**内部代码逻辑中直接调用预定义的函数或方法**，通常用于处理结构化数据、执行算法或完成特定子任务。
+ **特点**：
    - **<font style="color:rgb(51, 51, 51);">内部性</font>**<font style="color:rgb(51, 51, 51);">：</font>**<font style="color:#ED740C;">在Agent的代码或框架内实现，无需依赖外部系统。</font>**
    - **<font style="color:rgb(51, 51, 51);">同步执行</font>**<font style="color:rgb(51, 51, 51);">：直接返回结果，延迟低。</font>
    - **<font style="color:rgb(51, 51, 51);">结构化输入/输出</font>**<font style="color:rgb(51, 51, 51);">：参数和返回值通常是编程语言原生支持的数据类型（如字符串、列表、字典等）。</font>
    - **<font style="color:rgb(51, 51, 51);">确定性</font>**<font style="color:rgb(51, 51, 51);">：结果由内部逻辑决定，无需处理网络或外部服务的不确定性。</font>
+ **示例1**：  
一个聊天Agent调用内部函数 `parse_intent(user_input)` 解析用户意图，或调用 `generate_response(context)` 生成回复。
+ **示例2：**

```python
# 典型函数注册（OpenAI兼容格式）
functions = [
    {
        "name": "unit_converter",
        "description": "单位换算工具",
        "parameters": {
            "type": "object",
            "properties": {
                "value": {"type": "number"},
                "from_unit": {"type": "string"},
                "to_unit": {"type": "string"}
            }
        }
    }
]

# 调用过程
response = qwen.chat.completions.create(
    messages=[{"role": "user", "content": "10英里等于多少公里？"}],
    functions=functions,
    function_call="auto"
)

```

:::color5
**<font style="color:#601BDE;">2.Tool Call（工具调用）</font>**

:::

+ **定义**：  
指Agent通过**<font style="color:#ED740C;">调用外部工具或服务</font>**完成任务，例如访问API、数据库查询、调用第三方应用等。
+ **特点**：
    - **<font style="color:rgb(51, 51, 51);">外部性</font>**<font style="color:rgb(51, 51, 51);">：依赖外部系统（如HTTP API、数据库、硬件设备）。</font>
    - **<font style="color:rgb(51, 51, 51);">异步性</font>**<font style="color:rgb(51, 51, 51);">：可能需要等待网络响应，延迟较高。</font>
    - **<font style="color:rgb(51, 51, 51);">协议适配</font>**<font style="color:rgb(51, 51, 51);">：需遵循特定通信协议（如REST、gRPC），参数需序列化为协议支持的格式（如JSON）。</font>
    - **<font style="color:rgb(51, 51, 51);">容错处理</font>**<font style="color:rgb(51, 51, 51);">：需处理网络错误、超时、数据格式不一致等问题。</font>
+ **示例1**：  
Agent调用Google搜索API获取实时信息，或调用数据库接口查询用户订单状态。
+ **示例2：**

```python
# 工具定义（Qwen增强格式）
tools = [
    {
        "type": "function",
        "function": {
            "name": "flight_search",
            "description": "航班查询工具",
            "parameters": {...}
        }
    },
    {
        "type": "web_search",
        "config": {
            "api_endpoint": "https://search.api"
        }
    }
]

# 多工具协同调用
response = qwen.agent.run(
    query="找下周五杭州到北京最便宜的航班，并总结当地天气",
    tools=tools,
    execution_mode="parallel"  # 支持并行执行
)

```

:::color5
**<font style="color:#601BDE;">3.工作流程对比</font>**

:::

**<font style="color:rgb(51, 51, 51);">Function Call流程：</font>**

```python
用户输入 → 意图识别 → 选择函数 → 生成参数 → 执行函数 → 返回结果
```

**<font style="color:rgb(51, 51, 51);">Tool Call流程：</font>**

```python
用户输入
↓
任务分解（Task Decomposition）
↓
工具选择 → [工具1执行 → 结果解析] 
           [工具2执行 → 结果整合]
↓
多源验证（Cross-Check） 
↓
最终响应生成

```

**<font style="color:#601BDE;">4.应用场景对比</font>**

    1. <font style="color:rgb(51, 51, 51);">单位换算（数值计算）</font>
    2. <font style="color:rgb(51, 51, 51);">简单数据查询（天气/股票）</font>
    3. <font style="color:rgb(51, 51, 51);">基础格式转换（JSON→XML）</font>

# 复杂任务示例：旅行规划

周五晚出发，需要经济型酒店和当地美⻝推荐"

# 触发的工具链：
1. 日历工具 → 确认日期有效性
2. 酒店搜索API → 筛选符合预算的住宿
3. 餐饮推荐引擎 → 获取评分>4.5的餐厅
4. 路径规划服务 → 优化景点路线

**<font style="color:#601BDE;">3.协同工作示例</font>**

:::

1. **流程示例**：
    - <font style="color:rgb(51, 51, 51);">用户问：“今天北京的天气如何？”</font>
    - **<font style="color:rgb(51, 51, 51);">Function Call</font>**<font style="color:rgb(51, 51, 51);">：解析意图，提取关键词“北京”“天气”。</font>
    - **<font style="color:rgb(51, 51, 51);">Tool Call</font>**<font style="color:rgb(51, 51, 51);">：调用天气API获取数据。</font>
    - **<font style="color:rgb(51, 51, 51);">Function Call</font>**<font style="color:rgb(51, 51, 51);">：将API返回的JSON数据转换为自然语言回复。</font>
    - <font style="color:rgb(51, 51, 51);">输出：“北京今天晴，气温25℃。”</font>
2. **框架支持**：  
某些框架（如LangChain、AutoGPT）将两者统一抽象为“工具”，Agent通过决策模块（如LLM）选择调用内部函数或外部工具，例如：

```python
# LangChain示例：定义工具（可能是函数或API）
tools = [
    Tool(name="internal_parse", func=parse_intent, description="解析用户意图"),
    Tool(name="search_web", func=google_api_call, description="调用搜索引擎")
]
agent.run("用户问题", tools=tools)
```



## 函数调用(Function Call)
:::color3
**简介：**<font style="color:rgb(51, 51, 51);">Function Call（函数调用）是大语言模型（LLM）与外部工具/API交互的核心机制。它允许模型在对话过程中：</font>

+ <font style="color:rgb(51, 51, 51);">识别用户意图是否需要调用外部能力</font>
+ <font style="color:rgb(51, 51, 51);">结构化生成符合API要求的参数</font>
+ <font style="color:rgb(51, 51, 51);">触发预设的函数执行</font>
+ <font style="color:rgb(51, 51, 51);">将执行结果融入后续对话</font>

:::

## 工具调用(Tool Call)
:::color3
**简介：**<font style="color:rgb(51, 51, 51);">工具调用是大模型主动与外部程序或API交互的能力。当模型遇到需要实时数据、专业计算或无法直接生成的任务时，会通过预设接口调用外部工具，类似人类使用计算器处理复杂数学问题。</font>

:::

:::color5
**<font style="color:#601BDE;">1.工具调用示例</font>**

:::

<font style="color:rgb(51, 51, 51);">Qwen通过结构化输出实现工具调用：</font>

<font style="color:rgb(51, 51, 51);">模型自动生成这种结构化请求，开发者通过解析JSON调用对应工具。</font>

```json
{
  "tool_call": {
    "name": "weather_api",
    "parameters": {"city": "北京", "date": "2023-11-20"}
  }
}
```

:::color5
**<font style="color:#601BDE;">2.工作流程</font>**

:::

1. <font style="color:rgb(51, 51, 51);">意图识别：用户问"北京明天多少度？"</font>
2. <font style="color:rgb(51, 51, 51);">参数提取：解析出城市=北京，日期=明天</font>
3. <font style="color:rgb(51, 51, 51);">生成调用：输出上述JSON结构</font>
4. <font style="color:rgb(51, 51, 51);">执行工具：程序调用天气API</font>
5. <font style="color:rgb(51, 51, 51);">结果整合：将API返回的25℃插入回答模板</font>

:::color5
**<font style="color:#601BDE;">3.开发示例</font>**

:::

**<font style="color:#601BDE;">4.优点</font>**

:::

+ <font style="color:rgb(51, 51, 51);">突破模型固有知识限制（如实时汇率）</font>
+ <font style="color:rgb(51, 51, 51);">降低幻觉风险（依赖权威数据源）</font>
+ <font style="color:rgb(51, 51, 51);">提升处理精度（复杂计算交给专业工具）</font>
+ <font style="color:rgb(51, 51, 51);">支持业务系统集成（对接企业API）</font>

### <font style="color:rgb(51, 51, 51);">工具调用步骤</font>
1. **需求分析与工具选择**：
    - **<font style="color:rgb(51, 51, 51);">明确任务需求</font>**<font style="color:rgb(51, 51, 51);">：首先，明确智能体需要完成的任务。例如，智能体需要从互联网上搜索最新新闻信息。</font>
    - **<font style="color:rgb(51, 51, 51);">选择合适的工具</font>**<font style="color:rgb(51, 51, 51);">：根据任务需求，选择适合的工具或服务。比如，选择一个可靠的新闻API或者搜索引擎API。</font>
2. **工具集成**：
    - **<font style="color:rgb(51, 51, 51);">接口开发</font>**<font style="color:rgb(51, 51, 51);">：编写代码实现智能体与外部工具之间的接口。如果使用的是Python，可以使用</font>`<font style="color:rgb(51, 51, 51);">requests</font>`<font style="color:rgb(51, 51, 51);">库来发送HTTP请求；如果是Java，则可以使用</font>`<font style="color:rgb(51, 51, 51);">HttpClient</font>`<font style="color:rgb(51, 51, 51);">。</font>
    - **<font style="color:rgb(51, 51, 51);">工具配置</font>**<font style="color:rgb(51, 51, 51);">：配置工具的参数，比如API密钥、请求超时时间等，确保工具能够正确响应。</font>
3. **定义调用规则**：
    - **<font style="color:rgb(51, 51, 51);">触发条件</font>**<font style="color:rgb(51, 51, 51);">：设定工具调用的条件。例如，当用户查询“最新新闻”时，触发新闻搜索工具。</font>
    - **<font style="color:rgb(51, 51, 51);">调用顺序与参数设置</font>**<font style="color:rgb(51, 51, 51);">：定义工具调用的顺序，以及每次调用的参数值，以确保任务流程的正确性。</font>
4. **工具调用实现**：
    - **<font style="color:rgb(51, 51, 51);">编写调用代码</font>**<font style="color:rgb(51, 51, 51);">：根据工具的接口文档，编写具体的调用代码。例如，发送一个GET请求到新闻API，获取最新的新闻数据。</font>
    - **<font style="color:rgb(51, 51, 51);">错误处理</font>**<font style="color:rgb(51, 51, 51);">：实现错误捕捉和处理机制，确保在调用工具时，能够处理可能出现的网络超时、API错误等情况。</font>
5. **测试与优化**：
    - **<font style="color:rgb(51, 51, 51);">功能测试</font>**<font style="color:rgb(51, 51, 51);">：在不同的情景下测试工具调用的功能，确保智能体能够正确调用工具并获取预期的结果。</font>
    - **<font style="color:rgb(51, 51, 51);">性能优化</font>**<font style="color:rgb(51, 51, 51);">：根据测试结果，优化工具调用的效率，比如优化请求的频率，减少不必要的调用。</font>
    - **<font style="color:rgb(51, 51, 51);">错误修正</font>**<font style="color:rgb(51, 51, 51);">：根据测试中发现的问题，修正调用代码，确保系统稳定运行。</font>
6. **部署与监控**：
    - **<font style="color:rgb(51, 51, 51);">系统集成</font>**<font style="color:rgb(51, 51, 51);">：将工具调用功能集成到整个智能体系统中，确保与其他模块协同工作。</font>
    - **<font style="color:rgb(51, 51, 51);">监控与维护</font>**<font style="color:rgb(51, 51, 51);">：部署后，持续监控工具调用的性能和稳定性，及时发现并解决问题。</font>

### NL2SQL工具调用
<font style="color:rgb(51, 51, 51);">1. 确定任务需求和目标</font>

<font style="color:rgb(51, 51, 51);">首先，明确具体的任务目标。例如，一个NL2SQL系统需要将用户输入的自然语言查询转换为对应的SQL语句。在这个过程中，可能需要进行以下步骤：</font>

+ **<font style="color:rgb(51, 51, 51);">自然语言理解（NLU）</font>**<font style="color:rgb(51, 51, 51);">：解析用户的查询意图和需求。</font>
+ **<font style="color:rgb(51, 51, 51);">信息抽取</font>**<font style="color:rgb(51, 51, 51);">：识别查询中的实体、关系和属性。</font>
+ **<font style="color:rgb(51, 51, 51);">语义解析</font>**<font style="color:rgb(51, 51, 51);">：将这些信息与数据库模式匹配，生成相应的查询结构。</font>
+ **<font style="color:rgb(51, 51, 51);">SQL生成</font>**<font style="color:rgb(51, 51, 51);">：根据语义解析的结果，生成准确的SQL语句。</font>

<font style="color:rgb(51, 51, 51);">在这个过程中，某些步骤可能会比较复杂或者需要依赖外部工具来提高准确性和效率。例如，实体识别、语义解析和SQL优化等任务，可以借助现有的工具或服务来实现。</font>

<font style="color:rgb(51, 51, 51);">2. 选择适用的工具或服务</font>

<font style="color:rgb(51, 51, 51);">根据具体的任务需求，选择适合的外部工具或服务。例如：</font>

+ **<font style="color:rgb(51, 51, 51);">NLP工具</font>**<font style="color:rgb(51, 51, 51);">：如spaCy或HanLP，用于进行分词、实体识别和句法分析。</font>
+ **<font style="color:rgb(51, 51, 51);">语义解析工具</font>**<font style="color:rgb(51, 51, 51);">：如SQLExecutor，用于验证和优化生成的SQL语句。</font>
+ **<font style="color:rgb(51, 51, 51);">数据库交互工具</font>**<font style="color:rgb(51, 51, 51);">：如JDBC驱动或ORM框架，用于执行SQL查询并获取结果。</font>

<font style="color:rgb(51, 51, 51);">选择工具时，应考虑以下因素：</font>

+ **<font style="color:rgb(51, 51, 51);">功能匹配</font>**<font style="color:rgb(51, 51, 51);">：工具是否能够满足特定任务的需求。</font>
+ **<font style="color:rgb(51, 51, 51);">性能</font>**<font style="color:rgb(51, 51, 51);">：工具是否高效，能够处理预期的负载。</font>
+ **<font style="color:rgb(51, 51, 51);">兼容性</font>**<font style="color:rgb(51, 51, 51);">：工具是否与现有的系统架构和编程语言兼容。</font>
+ **<font style="color:rgb(51, 51, 51);">可扩展性</font>**<font style="color:rgb(51, 51, 51);">：工具是否能够随着系统的需求增长而扩展。</font>

<font style="color:rgb(51, 51, 51);">3. 设计工具接口</font>

<font style="color:rgb(51, 51, 51);">在将工具集成到NL2SQL系统之前，需要设计工具接口，确保系统能够有效调用这些工具。</font>

+ **定义输入输出格式**：工具的输入和输出数据格式需要与系统的其他部分兼容。例如，实体识别工具应该输出结构化的结果，以便后面的步骤进行处理。
+ **接口标准化**：如果使用多个工具，确保它们的接口有一定的标准化，方便集成和管理。



<font style="color:rgb(51, 51, 51);">4. 集成工具到NL2SQL系统</font>

<font style="color:rgb(51, 51, 51);">将编写好的工具调用代码整合到整个NL2SQL系统中，确保各个组件协同工作。</font>

+ **模块化设计**：
    - 将工具的调用封装成独立的模块或服务，方便管理和维护。
    - 使用依赖注入（Dependency Injection）等设计模式，提高系统的灵活性和可测试性。
+ **流程控制**：
    - <font style="color:rgb(51, 51, 51);">定义整个NL2SQL任务的流程，确定工具调用的触发条件和顺序。例如，在解析用户的查询之后，先调用实体识别工具，再进行语义解析，最后生成SQL语句。</font>

<font style="color:rgb(51, 51, 51);">5. 定义工具调用规则</font>

<font style="color:rgb(51, 51, 51);">为了确保工具被恰当地使用，需要定义明确的调用规则。</font>

+ **触发条件**：
    - <font style="color:rgb(51, 51, 51);">在什么时候调用哪个工具。例如，当检测到查询中包含特定关键词时，触发语义解析工具。</font>
+ **参数设置**：
    - <font style="color:rgb(51, 51, 51);">为每个工具调用定义输入参数和配置，确保工具能够按照预期处理任务。例如，设置实体识别工具的敏感度级别。</font>
+ **调用策略**：
    - <font style="color:rgb(51, 51, 51);">确定工具调用的频率和限制，避免过度调用导致性能下降或成本增加。</font>

## <font style="color:rgb(51, 51, 51);">MCP (</font><font style="color:rgb(25, 27, 31);">Model Context Protocol</font><font style="color:rgb(51, 51, 51);">) </font><font style="color:#D22D8D;">（by草莓师姐）</font>
:::success
**<font style="color:rgb(51, 51, 51);">背景</font>**<font style="color:rgb(51, 51, 51);">：</font><font style="color:rgb(25, 27, 31);">为了克服手工 prompt 的局限性，许多 LLM 平台（如 OpenAI、Google）引入了 </font>`<font style="color:rgb(25, 27, 31);background-color:rgb(248, 248, 250);">function call</font>`<font style="color:rgb(25, 27, 31);"> 功能。 function call 也有其局限性，重点在于 </font>**<font style="color:#117CEE;">function call 平台依赖性强，不同 LLM 平台的 function call API 实现差异较大</font>**<font style="color:rgb(25, 27, 31);">。例如，OpenAI 的函数调用方式与 Google 的不兼容，开发者在切换模型时需要重写代码，增加了适配成本。除此之外，还有</font>**<font style="color:#117CEE;">安全性，交互性等问题。</font>**

<font style="color:rgba(0, 0, 0, 0.9);">所以，你会发现，</font>**<font style="color:#117CEE;">OpenAI只是首先提出了function call概念，至于怎么实现的，还是要各家自己来做</font>**<font style="color:rgba(0, 0, 0, 0.9);">。而MCP直接开放了协议，希望把这种</font>**<font style="color:#74B602;">各自为政的function call统一到一起，成为一个标准</font>**<font style="color:rgba(0, 0, 0, 0.9);">。</font><font style="color:#D22D8D;">（by草莓师姐）</font>

:::

:::color3
**简介：**<font style="color:rgb(25, 27, 31);">MCP （Model Context Protocol，模型上下文协议）定义了应用程序和 AI 模型之间交换上下文信息的方式。这使得开发者能够</font>**<font style="color:rgb(25, 27, 31);">以一致的方式将各种数据源、工具和功能连接到 AI 模型</font>**<font style="color:rgb(25, 27, 31);">（一个中间协议层），就像 </font>**<font style="color:#ED740C;">USB-C </font>**<font style="color:rgb(25, 27, 31);">让不同设备能够通过相同的接口连接一样。MCP 的目标是创建一个</font>**<font style="color:#ED740C;">通用标准，使 AI 应用程序的开发和集成变得更加简单和统一。</font>**MCP<font style="color:rgb(25, 27, 31);">为LLM提供了</font>**<font style="color:#ED740C;">一套标准化协议</font>**<font style="color:rgb(25, 27, 31);">，旨在为LLM方便调用各类数据源和工具</font>

**paper：**[Introducing the Model Context Protocol](https://link.zhihu.com/?target=https%3A//www.anthropic.com/news/model-context-protocol)<font style="color:rgb(25, 27, 31);"></font>

**<font style="color:#601BDE;">3.MCP和Function Call的区别与联系</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

<font style="color:rgba(0, 0, 0, 0.9);">MCP并非Function Calling的替代品，而是通过标准化协议解决了碎片化集成的问题，同时为Function Calling提供了更广阔的协作空间。两者的关系类似于</font>**<font style="color:#74B602;">“高速公路与汽车”——MCP是连接万物的基础设施，Function Calling则是灵活高效的交通工具</font>**<font style="color:rgba(0, 0, 0, 0.9);">，二者结合才能实现AI应用的规模化落地。</font>

+ **<font style="color:rgba(0, 0, 0, 0.9);">Function Call</font>**<font style="color:rgba(0, 0, 0, 0.9);">：</font>**<font style="color:rgb(0, 82, 255);">好比家里的电器遥控器</font>**<font style="color:rgba(0, 0, 0, 0.9);">，每个遥控器都得单独编接口才能控制相应电器。比如想让AI查询天气，就得跟第三方天气应用对接API接口，完成了之后，这东东也不能用来查其他的，就是特定用途。这就好比你对接了一款遥控器之后，想要控制灯光，还得再搞个电灯遥控器一样。</font>
+ **<font style="color:rgba(0, 0, 0, 0.9);">MCP</font>**<font style="color:rgba(0, 0, 0, 0.9);">：就是想做一套</font>**<font style="color:rgb(0, 82, 255);">全屋智能中枢</font>**<font style="color:rgba(0, 0, 0, 0.9);">，通过统一协议接入所有家电，只需要家电采用相同的协议接入即可，这样，用户要睡觉，那就要首先控制窗帘关闭，接着关灯，把空调调到睡眠模式，而这一切，只需要一个“MCP”万能遥控器搞定，再也不用一个个找遥控器了！</font>

**<font style="color:#601BDE;">4.MCP的优点？为什么使用MCP</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

1. **<font style="color:rgb(25, 27, 31);">没有 MCP 之前我们会怎么做</font>**<font style="color:rgb(25, 27, 31);">？</font>

<font style="color:rgb(25, 27, 31);">我们可能会人工从数据库中筛选或者使用工具检索可能需要的信息，手动的粘贴到 prompt 中。随着我们要</font>**<font style="color:#117CEE;">解决的问题越来越复杂，手工把信息引入到 prompt 中会变得越来越困难</font>**<font style="color:rgb(25, 27, 31);">。</font>

2. **<font style="color:rgb(25, 27, 31);">function call的局限</font>**

<font style="color:rgb(25, 27, 31);">为了克服手工 prompt 的局限性，许多 LLM 平台（如 OpenAI、Google）引入了 </font>`<font style="color:rgb(25, 27, 31);background-color:rgb(248, 248, 250);">function call</font>`<font style="color:rgb(25, 27, 31);"> 功能。这一机制允许模型在需要时调用预定义的函数来获取数据或执行操作，显著提升了自动化水平。</font>

<font style="color:rgb(25, 27, 31);">但是 function call 也有其局限性，重点在于 </font>**<font style="color:#117CEE;">function call 平台依赖性强，不同 LLM 平台的 function call API 实现差异较大</font>**<font style="color:rgb(25, 27, 31);">。例如，OpenAI 的函数调用方式与 Google 的不兼容，开发者在切换模型时需要重写代码，增加了适配成本。除此之外，还有</font>**<font style="color:#117CEE;">安全性，交互性等问题。</font>**

3. **<font style="color:rgb(25, 27, 31);">MCP的优点</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>
    - **<font style="color:rgb(25, 27, 31);">生态</font>**<font style="color:rgb(25, 27, 31);"> </font><font style="color:rgb(25, 27, 31);">- MCP 提供很多现成的插件，你的 AI 可以直接使用。</font>
    - **<font style="color:rgb(25, 27, 31);">统一性</font>**<font style="color:rgb(25, 27, 31);"> </font><font style="color:rgb(25, 27, 31);">- 不限制于特定的 AI 模型，任何支持 MCP 的模型都可以灵活切换。</font>
    - **<font style="color:rgb(25, 27, 31);">数据安全</font>**<font style="color:rgb(25, 27, 31);"> - 你的敏感数据留在自己的电脑上，不必全部上传。（因为我们可以自行设计接口确定传输哪些数据）</font>

<font style="color:rgb(25, 27, 31);">MCP 由三个核心组件构成：Host、Client 和 Server。让我们通过一个实际场景来理解这些组件如何协同工作：假设你正在使用 Claude Desktop (Host) 询问："我桌面上有哪些文档？"</font>

1. **<font style="color:rgb(25, 27, 31);">Host</font>**<font style="color:rgb(25, 27, 31);">：Claude Desktop 作为 Host，负责接收你的提问并与 Claude 模型交互。</font>
2. **<font style="color:rgb(25, 27, 31);">Client 客户端</font>**<font style="color:rgb(25, 27, 31);">：当 Claude 模型决定需要访问你的文件系统时，Host 中内置的 MCP Client 会被激活。这个 Client 负责与适当的 MCP Server 建立连接。</font>
    - <font style="color:rgb(25, 27, 31);">MCP client 首先从 MCP server 获取可用的工具列表。</font>
    - <font style="color:rgb(25, 27, 31);">将用户的查询连同工具描述通过 function calling 一起发送给 LLM。</font>
    - <font style="color:rgb(25, 27, 31);">LLM 决定是否需要使用工具以及使用哪些工具。</font>
    - <font style="color:rgb(25, 27, 31);">如果需要使用工具，MCP client 会通过 MCP server 执行相应的工具调用。</font>
    - <font style="color:rgb(25, 27, 31);">工具调用的结果会被发送回 LLM。</font>
    - <font style="color:rgb(25, 27, 31);">LLM 基于所有信息生成自然语言响应。</font>
    - <font style="color:rgb(25, 27, 31);">最后将响应展示给用户。</font><font style="color:#D22D8D;">（by草莓师姐）</font>
3. **<font style="color:rgb(25, 27, 31);">Server 服务端</font>**<font style="color:rgb(25, 27, 31);">：在这个例子中，文件系统 MCP Server 会被调用。它负责执行实际的文件扫描操作，访问你的桌面目录，并返回找到的文档列表。MCP server 是 MCP 架构中的关键组件，它可以提供 3 种主要类型的功能：</font>
    - <font style="color:rgb(25, 27, 31);">资源（Resources）：类似文件的数据，可以被客户端读取，如 API 响应或文件内容。</font>
    - <font style="color:rgb(25, 27, 31);">工具（Tools）：可以被 LLM 调用的函数（需要用户批准）。</font>
    - <font style="color:rgb(25, 27, 31);">提示（Prompts）：预先编写的模板，帮助用户完成特定任务。</font><font style="color:#D22D8D;">（by草莓师姐）</font>

# A2A（Agent2Agent协议） <font style="color:#D22D8D;">（by草莓师姐）</font>

**<font style="color:rgb(51, 51, 51);">背景</font>**<font style="color:rgb(51, 51, 51);">：</font> AI 智能体（Agent）是越来越多了，不光 Agent 背后的 [LLM](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=LLM&zhida_source=entity) 琳琅满目，就连做 Agent 的框架数起来也跟报菜名差不多（ [LangGraph](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=LangGraph&zhida_source=entity)、[CrewAI](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=CrewAI&zhida_source=entity)、Google ADK、[Genkit](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=Genkit&zhida_source=entity)、[Semantic Kernel](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=Semantic+Kernel&zhida_source=entity)……），它们往往**由不同厂商提供，彼此之间缺乏统一的交互方式，很难协同工作**。例如，一个智能体负责处理文本查询，另一个智能体专注于图像生成，还有的智能体能完成财务审批或表单处理。这些 Agent 若想协作，需要频繁地「翻译」或定制适配层。

<font style="color:rgba(0, 0, 0, 0.9);">2025年4月9日，Google推出了一个名为 Agent2Agent （A2A） 的新开放协议，得到了 50 多个技术合作伙伴的支持和贡献，这些合作伙伴包括 Atlassian、Box、Cohere、Intuit、Langchain、MongoDB、PayPal、Salesforce、SAP、ServiceNow、UKG 和 Workday；以及领先的服务提供商，包括埃森哲、BCG、凯捷、Cognizant、德勤、HCLTech、印孚瑟斯、毕马威、麦肯锡、普华永道、TCS 和 Wipro。A2A 协议将允许 AI 代理相互通信、安全地交换信息以及在各种企业平台或应用程序之上协调作。</font><font style="color:#D22D8D;">（by草莓师姐）</font>

**简介：**Google 推出了 A2A（Agent2Agent）协议，为**<font style="color:#ED740C;">异构 AI 智能体之间的互通与交互提供通用的语言</font>**：

+ 能力发现：让不同厂家的 Agent 可以相互「发现」和「了解」对方的能力。
+ 互相交流：让 Agent 可以通过简单的 JSON-RPC 格式交换消息、管理任务、上传下载文件，以及进行更复杂的多轮对话或多人协作。
+ 安全与信任: 在企业环境中，让不同的 Agent 系统安全地交互至关重要。需要标准化的认证和授权机制来确保只有可信的 Agent 才能进行通信和访问资源。

只有具备了这些基础，许多现实世界的复杂任务编排 才能得以解决——这些任务往往需要多个 Agent 协同工作，每个 Agent 贡献其独特的专长。<font style="color:#D22D8D;">（by草莓师姐）</font>

## 为什么需要A2A
:::color5
**<font style="color:#601BDE;">1.A2A和MCP的关系 & 对比？</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

![](https://cdn.nlark.com/yuque/0/2025/png/29769680/1744535746518-dffdff94-ba45-42dc-a757-2c4b2dca02e9.png)

1. **架构对比**
+ **MCP：模块化的客户端-服务器架构。**MCP 由 Anthropic 提出，其核心是建立一种标准化的**客户端-服务器**架构。典型架构包括三部分：MCP Host（承载 AI 模型的应用，如对话界面）、MCP Client（主机应用中的“连接器”模块）和 MCP Server（外部数据源或工具的提供者）。当用户向 AI 提问或AI需要外部信息时，MCP Client 会代表 AI 模型向相应的 MCP Server 发出请求，Server 与后端的数据源或工具交互后，将结果返回给 Client，供模型使用。
+ **A2A：面向多代理协作的任务型架构。**A2A 是 Google 与50多家合作伙伴联合推出的开放协议，旨在让来自不同厂商或框架的 AI **代理（Agent）彼此直接通信、协同完成任务 **。在架构上，A2A并没有集中式的服务器节点，而是以代理对代理的点对点交互为核心。
2. **应用对比**
+ MCP 解决的是**<font style="color:#74B602;"> Agent 和工具/资源</font>**之间的交互问题，可以看做是一个应用商店协议，主要关注单个 Agent 内容如何更好的使用外部工具；
+ 而 [A2A 协议](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=A2A+%E5%8D%8F%E8%AE%AE&zhida_source=entity)解决的是 **<font style="color:#74B602;">Agent 和 Agent </font>**之间的交互问题，主要关注不同的 Agent 之间怎么协作的问题。

它们不是竞争关系，是**<font style="color:#74B602;">互补</font>**的。

**<font style="color:rgb(25, 27, 31);">互补关系</font>**<font style="color:rgb(25, 27, 31);">：A2A “补充 Anthropic 的 MCP，提供有用的工具和上下文给智能体”。例如：</font>

+ <font style="color:rgb(25, 27, 31);">MCP 确保智能体能访问数据和工具（如通过 Google Drive、Slack、GitHub 等）。</font>
+ <font style="color:rgb(25, 27, 31);">A2A 则让智能体能协作处理这些数据，完成任务。</font>

**案例分析：**

+ **<font style="color:rgb(25, 27, 31);">现实场景</font>**<font style="color:rgb(25, 27, 31);">：维修店员工使用专用工具（如千斤顶、万用表、套筒扳手）诊断和修复故障。他们常需处理陌生问题，过程中涉及与客户沟通、研究解决方案，以及与零件供应商协作。</font>
+ **<font style="color:rgb(25, 27, 31);">智能体映射</font>**<font style="color:rgb(25, 27, 31);">：</font>
    - **<font style="color:rgb(25, 27, 31);">MCP协议</font>**<font style="color:rgb(25, 27, 31);">：连接智能体与结构化工具（如“将平台升高2米”“将扳手向右旋转4毫米”）。</font>
    - **<font style="color:rgb(25, 27, 31);">A2A协议</font>**<font style="color:rgb(25, 27, 31);">：支持终端用户或其他智能体与维修工交互（如“我的车有异响”），实现动态对话（“请拍摄左轮照片”“发现液体渗漏，持续多久了？”），并协调零件供应商等外部智能体。</font>

:::color5
**<font style="color:#601BDE;">2.设计原则</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

+ **简单性 (Simple):** 尽可能复用成熟的现有标准，如 HTTP(S) 作为传输层，[JSON-RPC 2.0](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=JSON-RPC+2.0&zhida_source=entity) 作为消息格式，[Server-Sent Events](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=Server-Sent+Events&zhida_source=entity) (SSE) 用于流式传输。降低学习和实现的门槛。
+ **企业就绪 (Enterprise Ready):** 内置对认证、授权、安全、隐私、追踪和监控的考量。将 Agent 视为企业级应用，确保其在生产环境中的可靠性和安全性。
+ **异步优先 (Async First):** 协议设计优先考虑异步交互，以支持可能耗时很长（甚至需要数天）的任务和包含「人在回路」(Human-in-the-loop) 的场景。支持轮询、流式更新和推送通知等多种异步模式。
+ **模态无关 (Modality Agnostic):** 不限制 Agent 交互的内容类型。原生支持文本 (text)、文件 (file)、结构化数据 (data/forms)，并为未来支持音频/视频流、iframe 等富媒体交互奠定基础。
+ **不透明执行 (Opaque Execution):** Agent 之间交互时，无需共享其内部的「思考过程」、具体的实现逻辑或使用的工具 (Tools)。交互聚焦于上下文、状态、指令和数据，保护了 Agent 的内部实现细节和知识产权。

:::color5
**<font style="color:#601BDE;">3.目标</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

+ **统一的消息格式**<font style="color:rgb(51, 51, 51);">：基于 JSON-RPC 2.0，让请求和响应的结构得到通用且被广泛支持的 JSON 格式封装。</font>
+ **面向任务的抽象**<font style="color:rgb(51, 51, 51);">：提供了 Task（任务）的概念，让客户端和 Agent（作为服务器）围绕 Task 进行多轮会话，并且能上报中间结果和最终结果。</font>
+ **多种交互模式**<font style="color:rgb(51, 51, 51);">：支持文本、文件、JSON 结构化数据等多种内容，并可扩展到音视频流。</font>
+ **可扩展的安全认证**<font style="color:rgb(51, 51, 51);">：每个 Agent 都可以在 Agent Card 中声明自己的认证需求（如 </font>[OAuth2](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=OAuth2&zhida_source=entity)<font style="color:rgb(51, 51, 51);">、</font>[API Key](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=API+Key&zhida_source=entity)<font style="color:rgb(51, 51, 51);">、</font>[JWT](https://zhida.zhihu.com/search?content_id=722211426&content_type=Answer&match_order=1&q=JWT&zhida_source=entity)<font style="color:rgb(51, 51, 51);"> 等），让企业轻松整合身份管理。</font>
+ **灵活的推送与流式更新**<font style="color:rgb(51, 51, 51);">：通过 SSE（Server-Sent Events）或 webhook 推送，使长时间运行的任务或多轮对话能进行实时更新，简化客户端轮询逻辑。</font>

:::color5
**<font style="color:#601BDE;">4.通信协议</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

A2A 协议使用 **HTTP/HTTPS** 作为传输层，采用 **JSON-RPC 2.0** 进行消息体封装。一个典型的调用过程是：

+ 客户端发送一个带有 `jsonrpc`、`method`、`params` 等字段的 HTTP 请求到 Agent 的服务端 URL。
+ 服务端解析方法（如 `tasks/send`），执行相应逻辑后返回结果，如果是异步、长任务可能部分内容会通过 SSE 或推送通知进一步发送到客户端。

由于 JSON-RPC 和 HTTP 均是非常成熟的技术，任何编程语言都能快速实现非常轻量的 A2A 客户端或服务器端模块。这也是该协议非常「可移植、可扩展」的一个关键。

**协议核心组件**

+ **Agent Card**：一个公开的 JSON 文件（通常托管在 `/.well-known/agent.json`），用于描述该 Agent 的名称、功能、技能（Skill）、URL、认证方式等信息，便于客户端进行「服务发现」和「能力匹配」。
+ **Task**：表示一个具体的工作单元，具有唯一 ID，并可以在多轮交互中不断更新状态。
+ **Message**：客户端和 Agent 之间互通时用的消息对象（"user" 或 "agent" 角色），其中可以包含多种类型的 **Part**（如文本部分、文件部分、数据部分等）
+ **Artifact**：由 Agent 在执行任务过程中生成的输出结果。它与 Message 的差别在于，Artifact 通常是「结果物」或产物，而 Message 常用于「对话或指令」。
+ **Push Notification**：可选功能，如果 Agent 支持 `pushNotifications`，就可以向客户端指定的 URL 主动发起任务进度更新，而无需客户端轮询。
+ **Streaming**：如果 Agent 支持 `streaming` 功能，就可以在处理某个任务时，通过 `tasks/sendSubscribe` 使用 SSE 进行分段或实时地输出状态与结果。

A2A 协议定义了三个主要角色：

+ **用户 (User):** 最终使用 Agent 系统来完成任务的人或服务。

+ **远程 Agent (Server):** 实现了 A2A 协议接口、接收请求并执行任务的不透明（「黑盒」）Agent。它提供 A2A 服务。

**<font style="color:#601BDE;">3.</font>**`**<font style="color:#601BDE;">AgentCard</font>**`<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

`AgentCard` 是 A2A 中用于 **Agent 发现** 的关键机制。它是一个公开的 **JSON** 元数据文件，描述了一个 A2A Server Agent 的基本信息、能力、技能、API 端点以及所需的认证方式。

Client 如何找到一个 Agent 的 Agent Card？

+ **推荐标准:** A2A 推荐将 Agent Card 托管在 Agent 服务的基础 URL 下的一个 **熟知位置 (Well-Known URI)** ：`https://<agent-base-url>/.well-known/agent.json`。这类似于许多 Web 标准（如 OpenID Connect Discovery）的做法。Client 可以通过 DNS 找到 Agent 的服务器，然后发送 HTTP GET 请求到这个熟知路径来获取 Agent Card。
+ **其他方式:** 也可以通过私有注册中心 (Registry)、Agent 目录 (Catalog) 或市场 (Marketplace) 来发现和管理 Agent Card。

![](https://cdn.nlark.com/yuque/0/2025/tif/29769680/1744531984269-2117b6b6-1400-451b-b8df-722f158822f0.tif?x-oss-process=image/format,png)

<font style="color:rgb(25, 27, 31);">Client 通过解析 </font>`<font style="color:rgb(25, 27, 31);background-color:rgb(248, 248, 250);">AgentCard</font>`<font style="color:rgb(25, 27, 31);">，就能了解目标 Agent 能做什么、如何与之通信（URL）、是否支持流式传输或推送通知、需要哪种认证方式，以及它具体提供了哪些技能和这些技能的细节。</font>

**<font style="color:#601BDE;">4.工作的核心单元：Task</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

`<font style="color:rgb(25, 27, 31);">Task</font>`<font style="color:rgb(25, 27, 31);"> 是 A2A 协议中</font>**工作流的核心单元**<font style="color:rgb(25, 27, 31);">。它代表了 Client 请求 Server 完成的一项具体工作，并贯穿整个交互过程。</font>

+ **创建:**<font style="color:rgb(25, 27, 31);"> Task 总是由 Client 发起（通过 </font>`<font style="color:rgb(25, 27, 31);">tasks/send</font>`<font style="color:rgb(25, 27, 31);"> 或 </font>`<font style="color:rgb(25, 27, 31);">tasks/sendSubscribe</font>`<font style="color:rgb(25, 27, 31);"> 方法）。</font>
+ **状态管理:**<font style="color:rgb(25, 27, 31);"> Task 的状态由 Server 维护和更新。</font>
+ **唯一标识:**<font style="color:rgb(25, 27, 31);"> 每个 Task 都有一个由 Client 生成的 </font>**唯一 ID (**`**id**`**)**<font style="color:rgb(25, 27, 31);"> ，用于在后续的交互（如获取状态、发送补充信息、取消任务）中引用同一个任务。</font>
+ **会话关联:**<font style="color:rgb(25, 27, 31);"> Client 可以选择性地为 Task 设置一个 </font>`<font style="color:rgb(25, 27, 31);">sessionId</font>`<font style="color:rgb(25, 27, 31);">，将多个相关的 Task 归属于同一个会话中。</font>
+ **内容交换:**<font style="color:rgb(25, 27, 31);"> Client 和 Server 在 Task 的上下文中交换 </font>`<font style="color:rgb(25, 27, 31);">Message</font>`<font style="color:rgb(25, 27, 31);">。</font>
+ **结果产出:**<font style="color:rgb(25, 27, 31);"> Server 在 Task 执行过程中或完成后生成 </font>`<font style="color:rgb(25, 27, 31);">Artifact</font>`<font style="color:rgb(25, 27, 31);"> 作为结果。</font>

**<font style="color:#601BDE;">5.信息的载体：Message 与 Part</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

`Message` 和 `Part` 是 A2A 中实际**承载内容**的基本单位。`Part` 的设计体现了 A2A 的**模态无关性**。通过组合不同的 `Part`，可以构建复杂的多模态消息，例如，一个包含文本指令 (`TextPart`) 和相关图片 (`FilePart`) 的用户请求。

+ `**Message**`**:** 代表 Client 和 Agent 之间的一次**通信回合 (turn)**。它可以是用户的请求、Agent 的指令、状态更新、错误信息或任何非最终结果的内容。
+ `**Part**`**:** 是 `Message` 或 `Artifact` 内部包含的具体**内容片段**。一个 `Message` 或 `Artifact` 可以由一个或多个 `Part` 组成，每个 `Part` 可以有不同的类型和内容。

Message 结构如下：

```json
interface Message {
  role: "user" | "agent";       // 消息发送者的角色
  parts: Part[];                // 包含的实际内容部分列表 (至少一个)
  metadata?: Record<string, any> | null; // 与消息相关的任意元数据
}
```

`Part` 是一个联合类型 (Union Type)，可以是以下几种具体类型之一：

+ `**TextPart**`**:** 包含纯文本内容。
+ `**FilePart**`**:** 包含文件数据。文件内容可以通过两种方式提供： 
    - `**bytes**`**:** Base64 编码的文件内容（适用于较小文件）。
    - `**uri**`**:** 指向文件资源的 URI（适用于较大文件或外部存储的文件）。两者**互斥**。
+ `**DataPart**`**:** 包含结构化的 JSON 数据。常用于传输表单数据、配置信息或需要精确结构的输入/输出。

:::color5
**<font style="color:#601BDE;">5.任务的成果：Artifacft</font>**<font style="color:#D22D8D;">（by草莓师姐）</font>

:::

`Artifact` 代表 Agent 在执行 Task 过程中或完成时**生成的产物**。它可以是文档、图片、代码、结构化数据报告、API 调用结果等任何形式的输出。

+ **包含内容:** 与 `Message` 类似，`Artifact` 也由一个或多个 `Part` 组成。
+ **不可变性:** 一旦生成，`Artifact` 通常被认为是不可变的（但在流式传输中可以追加内容）。
+ **多个产物:** 一个 Task 可以生成多个 `Artifact`。例如，「生成一个网站报告」的任务可能产生一个包含文本分析的 `Artifact` 和一个包含图表图片的 `Artifact`。

```plain
interface Artifact {
  name?: string | null;          // Artifact 的名称 (可选, 例如："analysis_report.pdf")
  description?: string | null;   // Artifact 的描述 (可选)
  parts: Part[];                // 包含的实际内容部分列表
  // --- 流式传输相关字段 (可选) ---
  index?: number;                // Artifact 在流中的索引 (默认为 0)，用于标识和更新同一个 Artifact
  append?: boolean;              // 是否将此 Artifact 的 parts 追加到具有相同 index 的现有 Artifact 上
  lastChunk?: boolean;           // 是否是具有相同 index 的 Artifact 的最后一块数据
  // --- 元数据 ---
  metadata?: Record<string, any> | null; // 与 Artifact 相关的任意元数据
}
```

`index`, `append`, `lastChunk` 这三个字段主要用于 **流式传输 (**`**tasks/sendSubscribe**`**)** ，允许 Agent 逐步构建和发送大型 `Artifact`。例如，Agent 可以先发送一个包含部分文本的 `Artifact` (index=0, append=false)，然后发送后续文本块作为 `Artifact` (index=0, append=true)，最后发送包含最后文本块的 `Artifact` (index=0, append=true, lastChunk=true)。

**<font style="color:#601BDE;">1.技术创新点</font>**

1. **多智能体系统**：采用分层架构：
    - **基础智能体**：BaseAgent（状态管理）、ReActAgent（思考-行动循环）、ToolCallAgent（工具调用）；
    - **专业化智能体**：Manus（通用任务处理）、PlanningAgent（任务拆解与进度跟踪）、SWEAgent（代码工程）；
2. **ReAct 执行模式**：结合推理与行动，通过「思考（分析状态）→ 行动（调用工具）→ 步骤记录」的循环完成任务，例如处理 "分析网站SEO" 指令时，会依次执行网络抓取、数据清洗和报告生成；

    - [**BrowserUseTool**](https://zhida.zhihu.com/search?content_id=716873562&content_type=Answer&match_order=1&q=BrowserUseTool&zhida_source=entity)：模拟人类浏览器操作（滚动、点击）；
    - **GoogleSearch**：实时网络信息检索。

:::color5
**<font style="color:#601BDE;">2.未来优化点</font>**

:::

+ 搜索工具优化：默认实现的搜索工具是google搜索，在国内无法使用，可以考虑优化成百度搜索。比如：`pip install baidusearch`
+ 执行 Python 代码工具优化：默认是使用exec实现的，存在安全行问题，可以考虑使用沙箱环境或者容器执行python程序。比如：`pip install jupyter-client` 或者 `pip install PandoraBox`
+ 判断任务执行结束逻辑优化：默认使用terminate工具判断任务是否结束，没有专门的逻辑判断实现。
+ 要运行该agent必须依赖支持函数调用的模型，可以考虑扩展实现不支持函数调用的模型
+ 工具调用的prompt可以考虑动态注册，而不是固定写死，比如manus中的next_step_prompt中的工具就是固定的。

1. **<font style="background-color:#D9EAFC;">Reasoning</font>**

<font style="color:rgb(25, 27, 31);">ReAct 推理环节主要用于确定下一步执行的动作，在 OpenManus 中，下一步动作会从下面几个可选动作中进行选择：</font>

+ <font style="color:rgb(25, 27, 31);">PythonExecute() ：执行 Python 代码</font>
+ <font style="color:rgb(25, 27, 31);">GoogleSearch() ：执行 Google 搜索</font>
+ <font style="color:rgb(25, 27, 31);">BrowserUseTool() ：执行浏览器操作</font>
+ <font style="color:rgb(25, 27, 31);">FileSaver() ：保存文件</font>
+ <font style="color:rgb(25, 27, 31);">Terminate() ：终止</font>

**<font style="color:#ED740C;">如何基于大模型确定具体执行的动作呢</font>**<font style="color:rgb(25, 27, 31);">？主要基于 OpenAI 提供的 </font>[标准方案](https://link.zhihu.com/?target=https%3A//platform.openai.com/docs/api-reference/chat/create)<font style="color:rgb(25, 27, 31);"> ，通过 </font>`<font style="color:rgb(25, 27, 31);background-color:rgb(248, 248, 250);">tools</font>`<font style="color:rgb(25, 27, 31);"> 使用标准格式</font>**<font style="color:#74B602;">传入所有可选的工具列表</font>**<font style="color:rgb(25, 27, 31);">，之后</font>**<font style="color:#74B602;">大模型会根据当前的上下文选择一个最合适的工具</font>**<font style="color:rgb(25, 27, 31);">。对应的实现如下所示：</font>

```python
async def think(self) -> bool:
    if self.next_step_prompt:
        user_msg = Message.user_message(self.next_step_prompt)
        self.messages += [user_msg]

    # 调用大模型，传入所有可选的工具列表，之后大模型会根据当前的上下文选择一个最合适的工具

    response = await self.llm.ask_tool(
        messages=self.messages,
        system_msgs=[Message.system_message(self.system_prompt)]
        if self.system_prompt
        else None,
        tools=self.available_tools.to_params(),
        tool_choice=self.tool_choices,
    )
    self.tool_calls = response.tool_calls
```

2. **<font style="background-color:#D9EAFC;">Acting（行动）</font>**

<font style="color:rgb(25, 27, 31);">行动环节主要用于执行具体的动作，在 OpenManus 中，执行动作的实现如下所示：</font>

```python
async def act(self) -> str:
    results = []
    # 执行推理环节确定的所有工具调用

    for command in self.tool_calls:
        result = await self.execute_tool(command)

        # 将工具调用的结果保存起来，对应于上面的流程中的观察反馈

        tool_msg = Message.tool_message(
            content=result, tool_call_id=command.id, name=command.function.name
        )
        self.memory.add_message(tool_msg)
        results.append(result)

    return "\n\n".join(results)
```

## AI For BI
### 评测数据集
:::color5
**<font style="color:#601BDE;">1.数据集设计原则</font>**

:::

+ **正向数据集设计原则**：根据大模型质量分级标准，针对不同级别的提问、覆盖评测场景中每一个指标组
+ **反向数据集设计原则**：过往日志中明显的badcase放到评测集中，命中该数据集越多，说明质量越不好
+ **需求为主**<font style="color:rgb(25, 27, 31);">：</font>从需求出发，能验证大模型回答内容的有效性（边界值分析、等价划分法、因果图法、错误推断法等）
+ **场景化**<font style="color:rgb(25, 27, 31);">：</font>根据不同的用户和场景，设计不同的评测集，比如商家小助手主要是评测意图识别后调用的API准确率，而驾驶舱场景主要是回答的文本中提取解读信息的能力
+ **异常处理**：异常场景的兜底回答是否符合常理、逻辑、安全、政治等（通用GPT打分实现）<font style="color:rgb(25, 27, 31);">；</font>明确不在需求范围内的case，放在反向评测集中
+ **评测case独立性**<font style="color:rgb(25, 27, 31);">：</font>从手工、历史数据、大模型生成三个渠道生成的评测集，需要按场景去重，针对一个case给出确定性答案

:::color5
**<font style="color:#601BDE;">2.测试场景分析</font>**

:::

通过需求评审与测试分析，大致确定可能的正向与逆向场景。

异常场景示例：

+ 意图模糊，查数还是找数，e.g. “商品"
+ 恶意，长周期需要缩短，e.g. “近10年的gmv"
+ 找数缺少时间周期，需补足 e.g. “销量是多少"

**<font style="color:#601BDE;">3.轨迹评估</font>**

:::

<font style="color:rgb(25, 27, 31);">应该从哪些方面评估Agent？怎样评估特定领域/用途的Agent？这两个问题目前还仍然没有得到很好的解决，但LangChain从</font>**<font style="color:rgb(25, 27, 31);">Trajectory（轨迹）</font>**<font style="color:rgb(25, 27, 31);">这个方向为这两个问题指出了一条路。</font>

<font style="color:rgb(25, 27, 31);">轨迹也就是Agent采取的一系列动作及其相应的响应，</font>**<font style="color:#74B602;">LangChain中的</font>**[**<font style="color:#74B602;">Trajectory Evaluators</font>**](https://zhida.zhihu.com/search?content_id=246478763&content_type=Article&match_order=1&q=Trajectory+Evaluators&zhida_source=entity)**<font style="color:#74B602;">（轨迹评估器）提供了一种更全面的方法来评估Agent的效率和能力</font>**<font style="color:rgb(25, 27, 31);">。简单来说，轨迹评估器获取了输入用户/Agent的输入、Agent的最终输出和中间步骤，然后传递给语言模型(默认为GPT-4)来对Agent输出的必要性、工具使用的合理性等进行评分和解释。</font>
